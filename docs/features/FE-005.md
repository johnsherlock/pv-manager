# FE-005: Direct MyEnergi Backend and Daily Summary APIs

## Goal

Replace the temporary V1 backend bridge with rewrite-owned backend services in
`apps/web`, using a low-cost Vercel-friendly deployment shape.

This feature covers three production backend capabilities:

- a direct MyEnergi-backed API for Live and single historical day reads
- a scheduled daily-summary job that runs just after each installation's local
  midnight and writes the finalized previous day
- a summary-backed range API that serves historical reporting from persisted
  `daily_summaries`

The purpose of this feature is to establish the rewrite's real backend spine so
later Overview, Range History, and multi-user work can build on app-owned
contracts rather than on the temporary V1 proxy and legacy Lambda estate.

## Feature Completion Criteria

- The rewrite can query MyEnergi directly for today or a requested historical
  local date without exposing provider details to the client
- The single-day provider adapter resolves local-day boundaries correctly across
  DST start and DST end transitions
- A secure scheduled job path exists for summarizing the previous completed day
  for all active installations
- Daily summary writes are idempotent and safe to retry
- The range API reads persisted `daily_summaries` and tariff data rather than
  calling the provider directly
- The range API returns completeness/trust and tariff-change metadata suitable
  for later History and Overview screens
- Verification exists for the direct adapter path, scheduled summarization, and
  first range-summary slice

## Notes

- This feature intentionally moves the rewrite from a temporary V1 bridge to a
  rewrite-owned backend shape while keeping the UI-facing contracts
  provider-agnostic.
- For the first beta, we can assume all supported users are in Ireland and all
  installations use `Europe/Dublin`. This allows one Vercel cron shortly after
  local midnight GMT/BST, while still requiring local-date logic to respect DST
  transitions correctly.
- For direct MyEnergi reads and summary generation alike, "the requested day"
  means the installation's local calendar day, not a naive fixed UTC window.
- Direct MyEnergi access must read credentials from the installation's stored
  provider connection context, not from one deployment-wide env var pair.
- The onboarding plan must capture MyEnergi credentials per user/installation so
  the backend feature can operate against real provider connections.
- `P-022` is pulled forward into this feature and re-scoped as the first
  production range-summary API rather than as a vague later billing read path.
