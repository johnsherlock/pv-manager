# P-021: Seed database with summary-first development fixture data

## Objective

Provide a repeatable way to populate the database with a small, coherent dataset
for local development and testing using the revised summary-first schema,
including the installation finance inputs needed for day 1 payback reporting.

## Scope

- Extend the installation schema if needed so financed installations can record payment duration for payback reporting
- Add a seed script inside `apps/web`
- Insert:
  - 1 user
  - 1 installation with realistic ownership or finance fields populated
  - 1 provider connection
  - 1 tariff plan with at least 2 versions (date-ranged)
  - a small set of `daily_summaries` spanning multiple days
- Data must exercise:
  - tariff version boundaries
  - summary-backed billing logic paths
  - installation payback inputs, including finance term when `finance_mode = finance`

## Out of Scope

- No UI work
- No external data ingestion
- No large datasets
- No persisted interval-reading fixtures unless a later story needs them

## Expected Implementation

- Seed script located at:
  - `apps/web/src/db/seed.ts` (or similar)
- Add npm script:
  - `pnpm db:seed` (or equivalent)

## Constraints

- Data must be:
  - deterministic
  - readable
  - small enough to inspect manually
- Fixture data should feel realistic enough to validate tariff transitions and period totals

## Dependencies

- Revised schema from P-020
- Domain logic from P-019

## Acceptance Criteria

- Running the seed command inserts a valid summary-first dataset
- If the current schema cannot represent financed-installation duration cleanly, P-021 includes the required schema update before seeding
- Data spans tariff version windows
- Data can be queried by the first billing slice without provider API calls
- No errors occur during seed execution

## Tests

- Optional: basic test verifying seed inserts expected entities and daily summary rows

## Notes

- This data will be used by P-022, the first summary-backed billing read path
- The seeded installation should include the finance/payback fields required by the day 1 "paid for itself" view
