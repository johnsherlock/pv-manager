# U-020 Mid-Fidelity Product Layout Spec
## Summary
Create an in-repo mid-fi design spec for the primary product surfaces so the team can move from low-fi structure to reviewable product layouts without jumping straight to high-fidelity mockups.

The U-020 output should cover the first-wave screen set only:

# Overview
- Live
- Daily History
- Range History
- Onboarding / setup
- Tariffs / contract management
- Data Health
- The deliverable should be a documentation artifact, not implementation or polished visual design. Its job is to lock layout hierarchy, module composition, copy structure, interaction rules, and state behavior so - U-021 through U-026 can proceed without re-deciding product structure.

# Key Changes
## New artifact
Add a new mid-fi spec document:

## docs/ui/mid-fi-layouts.md
That document should become the source of truth between low-fi wireframes and later high-fi work. It should translate the existing low-fi material from docs/ui/wireframes.md, docs/ui/screen-inventory.md, and docs/ui/data-contracts.md into reviewable screen-by-screen layouts.

## What each screen section should include
For each first-wave screen, define:

Screen purpose and primary question the screen answers
Desktop layout zones in reading order
Mobile layout zones in reading order
Required modules/cards shown on the screen
Module priority: primary, secondary, tertiary
Interaction notes: navigation, toggles, filters, date controls, edit entry points
State rules: loading, empty, stale, partial, setup-incomplete, hard error
Conditional rendering rules tied to missing tariff, finance, or installation-profile data
Content guidance: headline style, helper text, warning tone, CTA labels
Notes on what changed from low-fi, especially where layout hierarchy is clarified
Screen decisions to lock in U-020
Lock these decisions in the spec so later work does not drift:

Overview remains the default signed-in landing screen
Setup progress is a persistent product concept, not just an onboarding detail
Provider connection is the only required onboarding step for first usable value
Savings/payback never show fake zero states; they degrade to prompt cards when prerequisite data is missing
Live, Daily, and Range views should share visual language but keep distinct primary questions
Data Health is a dedicated surface, not only embedded warnings
Tariff management must include both history readability and edit flows
Installation date and array capacity are optional but materially improve efficiency and backfill UX

# Acceptance checks:

Every first-wave screen from the agreed scope has both desktop and mobile mid-fi layouts
Every layout clearly identifies primary vs secondary modules
Every layout explicitly handles setup-incomplete, stale, partial, and error states where applicable
Overview, Live, Daily, and Range can be distinguished by purpose in under a minute of review
Onboarding makes provider-first setup and optional later completion unmistakable
Tariff and Data Health screens show actionable trust/reminder behavior, not just raw information
The spec is detailed enough that U-021 high-fi work can proceed without guessing hierarchy or interactions
The spec stays aligned with the existing data-contract rules and does not invent backend requirements that conflict with them

# Review scenarios:

New user connects only provider details and lands on a useful Overview/Live experience
User without tariff info sees prompt cards instead of savings figures
User with missing installation date still understands backfill behavior
User updating tariff history can tell what is current, expired, or needs correction
User seeing stale live data understands both the warning and the next action

# Assumptions
U-020 should cover the primary product screens only, not every deferred/admin/later-stage surface
The deliverable should stay in-repo as a structured mid-fi spec, not require Figma to count as complete
U-020 is still a planning/design-definition story, not frontend implementation and not high-fidelity visual design
High-fidelity polish, prototype linking, design tokens, and implementation-ready specs remain in U-021 through U-026
Where low-fi wireframes contain chart/module ideas that feel too dense, U-020 should simplify in favor of decision clarity and trust rather than preserving every placeholder element
