# FE-009: Public and Unauthenticated Experience

## Goal

Define the non-signed-in surface of the product: how prospective users discover
the app, understand what it does, preview it safely, request beta access, and
move into the approval-gated account flow without confusion.

## Status

`Todo`

## Feature Completion Criteria

- A clear unauthenticated entry path exists for prospective beta users
- A read-only public demo mode exists so users can preview the app before
  signing in
- Beta access request flow is defined end to end
- Public-facing product and access messaging set correct expectations before
  sign-up
- The public surface can grow later into welcome, demo, and informational pages
  without reworking the auth foundation

## Included Stories

Stories in intended execution order:

1. [`P-012`](../backlog.md) Design beta access request and invite flow
2. [`U-014`](../backlog.md) Build beta access request form

No dedicated demo-mode backlog story exists yet, but demo mode is explicitly in
scope for this feature.

## Deferred From This Feature

- Marketing-site depth beyond what beta access needs
- Pricing or premium upsell treatment

## Notes

- This feature is deliberately small for now: it gives the unauthenticated
  product surface a home without forcing a full marketing-site project yet.
- The first demo mode should reuse seeded/sample data, stay strictly read-only,
  and hide Settings or any mutation-capable UI rather than exposing disabled
  edit controls.
