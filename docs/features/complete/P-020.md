# P-020: Align the rewrite schema with the summary-first v1 model

## Objective

Revise the rewrite app schema so the executable database model matches the
current v1 product direction:

- provider-backed live and single-day detail fetched on demand
- persisted `daily_summaries` for multi-day history
- no assumed requirement for persisted half-hour or minute interval rows

## Scope

- Review the current Drizzle schema and migration baseline in `apps/web`
- Decide the v1 role of:
  - `daily_summaries`
  - `provider_connections`
  - job and data-health tables
  - `energy_readings`
  - `billing_comparisons`
- Update schema code and migration plan so v1 storage is centered on:
  - users
  - installations
  - provider connections
  - tariff plans and versions
  - daily summaries
  - operational job / health state

## Out of Scope

- No provider API implementation
- No UI work
- No historical bill reconciliation workflow
- No interval persistence unless a concrete v1 need is identified during the review

## Expected Implementation

- Update schema definitions under:
  - `apps/web/src/db/schema.ts`
- Add a new migration or migration reset strategy under:
  - `apps/web/drizzle`
- Update any directly affected schema documentation if needed

## Constraints

- Keep the schema compatible with local development
- Prefer removing or deferring premature tables over keeping speculative ones
- Preserve a clean path to add interval persistence later if a real use case appears

## Dependencies

- Existing migration baseline from P-018
- Billing-oriented domain direction from P-019
- Decision record `0001`

## Acceptance Criteria

- The executable schema matches the summary-first v1 decision
- `daily_summaries` is a first-class persisted table for historical reporting
- The status of `energy_readings` and `billing_comparisons` is explicit:
  - removed from v1, or
  - retained with a clearly documented deferred role
- The schema remains coherent for local development and future server-side jobs

## Tests

- Migration generation or validation succeeds
- Schema changes do not break the existing rewrite test harness

## Notes

- This story exists to prevent implementation from building on the wrong runtime model
- Follow-on stories should seed and read from the revised schema rather than the earlier interval-first shape
