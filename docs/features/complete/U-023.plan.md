## Ambiguities and Assumptions

### Ambiguity 1 — What does "clickable" mean?

In a Figma context, clickable means hotspot navigation between frames. In a Next.js
prototype context, "clickable" could mean:

- (A) State switching — toggle between healthy / stale / warning variants within the
  same page
- (B) Navigation — clicking "View full day →" routes to another prototype screen

**Assumption:** State switching is the primary "clickable" requirement for U-023,
implemented as a `?state=healthy|stale|warning` URL search param so the state switcher
is a set of `<Link>` elements — purely server-side, no client JS needed for switching.
Navigation links (e.g. "View full day →") route to `#` for now and will be wired in
U-030 or later. This is the minimum that satisfies the acceptance criteria.

### Ambiguity 2 — Tailwind version

The CLAUDE.md specifies Tailwind CSS but not a version. shadcn/ui's current setup
tooling (`shadcn init`) targets **Tailwind CSS v3**; v4 compatibility is experimental
as of March 2026.

**Assumption:** Use **Tailwind CSS v3** (`tailwindcss@^3`, `postcss`,
`autoprefixer`). If the project later migrates to v4, that is a separate story.

### Ambiguity 3 — Colour system: Tailwind defaults vs existing palette

The existing `globals.css` defines a warm earthy palette using CSS variables
(`--accent: #0d6b57`, `--bg: #f3efe6`, etc.). The hi-fi spec defines semantic color
roles (`brand-primary`, `positive`, `warning`, `destructive`, `surface`, etc.).

**Assumption:** Map the hi-fi color roles onto the existing CSS variable palette and
extend Tailwind's theme to expose them as utility classes (e.g.
`bg-surface`, `text-on-surface`, `bg-brand-primary`). Do not wholesale replace the
existing palette with Tailwind defaults. This preserves the established visual tone and
means U-025 (token formalization) can later lock the values without a visual rework.

The mapping to start with:

| Hi-fi role | CSS variable / value |
|---|---|
| `surface` | `#f3efe6` (--bg) |
| `surface-raised` | `rgba(255,252,245,0.88)` (--panel) |
| `on-surface` | `#1f2a2a` (--text) |
| `on-surface-muted` | `#596868` (--muted) |
| `brand-primary` | `#0d6b57` (--accent) |
| `positive` | `#0d6b57` or a slightly adjusted green |
| `warning` | `#b45309` (amber-700 equivalent) |
| `destructive` | `#b91c1c` (red-700 equivalent) |
| `neutral` | `#374151` (gray-700 equivalent) |
| `border` | `rgba(31,42,42,0.12)` (--border) |

### Ambiguity 4 — Does the prototype route need auth gating?

**Assumption:** No. `/prototype/live` is publicly accessible in the dev environment.
It will be clearly labelled as a prototype with the `StateSelector` banner at the top.
Auth is a separate concern (U-015) and should not block prototype review.

### Ambiguity 5 — Recharts in a Server Component

`app/prototype/live/page.tsx` is a Server Component by default in the App Router.
Recharts requires a browser DOM environment.

**Assumption:** `LiveTrendChart.tsx` is a Client Component (`'use client'`). The page
renders all other content server-side and imports the chart as a client leaf. This is
standard Next.js practice and does not affect the prototype's ability to demonstrate
all three states via URL params.

### Ambiguity 6 — Is there a `docs/features/todo/U-023.md` file?

No. There is no feature file for U-023. **Assumption:** The plan is derived from the
backlog entry, the mid-fi layouts, the data contracts, and the hi-fi Live screen spec
produced in U-021 (available on branch `feature/U-021`). The plan captures all
decisions that would normally live in a feature file.

## Scoped Implementation Plan

### Step 1 — Create feature branch

```bash
git checkout main && git pull && git checkout -b feature/U-023
```

### Step 2 — Install the preferred UI stack

Install packages:

```
tailwindcss@^3, postcss, autoprefixer
recharts
lucide-react
clsx, tailwind-merge
```

shadcn/ui is initialized via its CLI (`npx shadcn@latest init`) which generates
`components.json`, updates `globals.css`, and sets up `lib/utils.ts`. Run from
`apps/web/`.

Then add the specific shadcn components needed for the prototype:

```
npx shadcn@latest add card badge button
```

### Step 3 — Configure Tailwind to use the existing palette

In `tailwind.config.ts`, extend the `theme.colors` and `theme.backgroundColor` to
expose the CSS variable palette as Tailwind utilities. Map the hi-fi semantic roles
listed in Ambiguity 3 above. Do not replace the existing `globals.css` CSS variable
declarations — Tailwind utilities will reference them.

Update `globals.css` to add the three Tailwind v3 directives (`@tailwind base`,
`@tailwind components`, `@tailwind utilities`) at the top while preserving all
existing custom rules.

### Step 4 — Define the mock data module

Create `apps/web/app/prototype/live/mock-data.ts`. Export three named objects, one per
state:

```
healthyState: LiveScreenData
staleState:   LiveScreenData
warningState: LiveScreenData
```

`LiveScreenData` is a locally-defined TypeScript interface (not shared with domain
layer — this is prototype-only). It captures everything the page needs to render
without any server calls:

- `trustState: 'fresh' | 'stale' | 'warning' | 'disconnected'`
- `freshnessLabel: string` (e.g. "Updated 12 seconds ago")
- `showWarningBanner: boolean`
- `warningMessage?: string`
- `metrics: { generation: number; consumption: number; import: number; export: number; unit: 'W' | 'kW' }`
- `coveragePercent: number`
- `trendData: Array<{ time: string; generation: number; consumption: number; import: number; export: number }>`
- `todayTotals: { generated: number; consumed: number; imported: number; exported: number }`
- `efficiencyPercent?: number` (omit from stale/warning to test conditional rendering)
- `notes?: string[]`

Stale state: same metric values but trust fields set to stale, warningBanner shown.
Warning state: set to provider-disconnected or severely stale; metrics shown as last-known.

### Step 5 — Build the prototype page

`apps/web/app/prototype/live/page.tsx` is a **Server Component**.

It reads `searchParams.state` (defaulting to `'healthy'`), selects the correct mock
data object, and renders the full page layout:

```
<StateSelector currentState={state} />         ← prototype chrome, outside main layout
<nav bar: "Live" + auto-refresh label + trust badge />
<WarningBanner />                               ← conditional
<section: 4-column MetricCard row />
<SolarCoverageBar />
<LiveTrendChart />                              ← client component
<section: 2-col — CurrentDayTotals + NotesPanel />
```

Desktop: 4-column KPI row, 2-column secondary row, implemented with CSS Grid or Tailwind
`grid` utilities. Mobile: single-column stack via Tailwind responsive breakpoints.

### Step 6 — Build each component

**StateSelector** (`prototype-only`):
- Horizontal segmented control fixed at top of viewport with a "PROTOTYPE" label
- Three links: `?state=healthy`, `?state=stale`, `?state=warning`
- Currently selected state is highlighted
- Uses `sticky` positioning above the main nav; visually distinct (e.g. dark bar) so
  it does not get confused with product chrome

**TrustBadge**:
- Props: `trustState`, `freshnessLabel`
- Renders a small pill with a coloured dot and label
- Colour roles: fresh → `brand-primary`, stale → `warning`, disconnected → `destructive`
- Uses shadcn `Badge` or a custom span with Tailwind utilities

**WarningBanner**:
- Props: `message`, `ctaLabel`, `ctaHref`
- Full-width bar; calm tone; `warning` background at low opacity; icon (lucide
  `AlertTriangle`) + message + CTA link
- Conditionally rendered; hidden when `showWarningBanner` is false

**MetricCard**:
- Props: `label`, `value`, `unit`, `supportText`, `stale?: boolean`
- Wraps shadcn `Card`; `display`-size value, `headline` label, `body-sm` support text
- When `stale=true`: desaturate the value text (`text-on-surface-muted`) and show a
  clock icon (lucide `Clock`) next to the value
- Used four times: Generation, Consumption, Import, Export

**SolarCoverageBar**:
- Props: `coveragePercent`, `stale?: boolean`
- Horizontal progress bar: `brand-primary` fill for solar, `neutral` for grid
- Percentage label below bar
- When stale: bar frozen, add `warning` border around the bar card

**LiveTrendChart** (`'use client'`):
- Props: `data: trendData[]`, `stale?: boolean`
- Recharts `AreaChart` inside `ResponsiveContainer`
- Four series: generation (brand-primary), consumption (neutral), import (neutral
  dimmer), export (positive)
- Time-resolution toggle: segmented control inside the chart card header (Day / 15min)
  — state is local to this client component; no URL sync needed in prototype
- When stale: desaturate all series fills, add annotation text on x-axis end:
  "Data delayed"
- Empty state: centered "No data" message in the plot area

**CurrentDayTotals**:
- Props: `totals: { generated, consumed, imported, exported }`
- Label + value pairs, `body` / `headline` rhythm
- "View full day →" link routes to `#`

**NotesPanel**:
- Props: `efficiencyPercent?: number`, `notes?: string[]`
- Efficiency line only when prop is present (tests silent omission)
- Notes list for partial-day or data-health messages

### Step 7 — Verify responsive layout

Open the prototype at `/prototype/live` in the browser. Check:
- Desktop (>1024px): 4-column KPI row, 2-column secondary row
- Tablet (768–1024px): 2-column KPI row, 1-column secondary
- Mobile (<768px): all cards stacked single-column

No automated tests are required for the prototype itself (it contains no domain logic).
The build must compile without TypeScript errors (`next build`).
