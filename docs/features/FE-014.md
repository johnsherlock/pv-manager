# FE-014: Remaining Setup Completion

## Goal

Finish the signed-in optional setup experience by implementing the remaining
Settings work that was intentionally left as placeholders after `FE-008`:
system investment history, Location, and Notifications.

This feature should turn the Settings home from a partly-complete roadmap into
a trustworthy product area where users can actually complete the setup steps
that unlock payback, yield context, weather/location context, and reminder
preferences.

Unlike `FE-008`, this feature should include the first meaningful downstream
product behaviour unlocked by those setup domains, especially around financial
payback interpretation, rather than stopping at edit forms alone.

The finance and physical-system setup model should no longer assume one flat
installation-level investment. The real target for this feature is a
multi-investment system history where users can capture the original install
and later expansions/upgrades as separate dated system-addition records, each
with their own physical and payment details.

## Status

`Todo`

## Feature Completion Criteria

- Users can manage real Settings flows for system additions/investments,
  Location, and tariff reminders instead of placeholder "coming soon" pages
- Each remaining Settings domain has a clear read/edit flow, completion signal,
  and explanation of what product value it unlocks
- Setup-completion progress on `/settings` reflects the true state of the
  optional setup domains
- Users can remove previously entered setup data, and completion state returns
  to incomplete when the remaining saved data no longer meets that domain's
  minimum requirements
- Users can capture more than one dated system-addition record for a single
  installation
- Each investment record can link together the physical addition and the way it
  was paid for without forcing users to model separate finance agreements and
  asset objects manually
- Investment inputs are sufficient to support trustworthy payback-related
  product behaviour across month, year, and lifetime-style views where the
  underlying data coverage allows it
- The first downstream finance surfaces can explain savings versus repayments,
  contribution toward investment recovery, and confidence limits when history
  is incomplete
- Physical system details are sufficient to support later solar efficiency and
  yield comparison work without rethinking the core setup model again
- Location setup supports the existing exact/approximate location approach used
  by Live weather and solar-context features
- Tariff-reminder preferences have a concrete user-facing home, even if the
  first implementation remains intentionally narrow

## Deferred From This Feature

- Public/demo-mode setup messaging and unauthenticated onboarding
- Admin/operator-only tools
- Broader marketing or beta-access flows
- Full Data Health coverage, outage-detection persistence, and outage email
  delivery, which remain part of `FE-002`
- Rich analytics or comparison surfaces beyond the first setup-unlocked finance
  and interpretation views
- Advanced notification delivery/reporting beyond the first concrete
  user-manageable reminder preference set
- Complex lender/accounting features such as refinancing history, APR
  breakdowns, tax treatment, or supplier-finance reconciliation

## Notes

- `FE-008` intentionally established the Settings shell and fully implemented
  Tariffs as the first vertical slice; this feature picks up the remaining
  Settings domains that were left as placeholders.
- This feature should stay focused on finishing setup and the first directly
  unlocked user value, not on every downstream analytics surface those inputs
  may later enable.
- Existing work on location capture, persistence, and Live weather context
  should be reused rather than redesigned here.
- Location should remain as simple as the weather and daylight context needs
  for now; exact geocoded precision is useful only when it materially improves
  those experiences.
- Reminder preferences belong here only insofar as users need a Settings home
  for tariff-validity reminders. Provider-outage alerting and anomaly emails
  depend on persisted coverage and incident logic, so they should stay aligned
  to `FE-002`.
- Setup domains in this feature should support intentional removal as well as
  creation and editing. Deleting the last meaningful record in a domain should
  return that domain to incomplete and update `/settings` progress accordingly.
- The first-pass flat finance model has already been built, but it should now
  be treated as an interim slice rather than the final product shape.
- The real target model is a list of dated system-addition records for one
  installation. Each record should represent a concrete step in the system's
  evolution, such as:
  - original install
  - rear-roof expansion
  - battery addition
- Each system-addition record should combine:
  - a human-readable label
  - a full date
  - the physical addition details relevant in the first pass
  - the payment details relevant to that addition
- The first-pass physical fields per system-addition record should stay minimal:
  - label / description
  - date
  - capacity added (`kWp`) when relevant
- The first-pass payment fields per system-addition record should stay minimal:
  - upfront payment
  - monthly repayment amount (interest included)
  - repayment duration in months
- Validation rules should stay simple:
  - label is required
  - one full date is required for all system-addition records
  - at least one of upfront payment or monthly repayment must be entered
  - if monthly repayment is entered, repayment duration is required
- `kWp` should stay optional in the first pass, but the record is still meant to
  represent one concrete system investment step rather than a finance-only note
- For this first pass, a system-addition record should always include usable
  finance details. If a user does not want to share investment details, they
  should not complete this domain yet.
- This keeps the model intentionally simple: adding system history here is
  specifically about unlocking ROI and system-performance interpretation
  together, not storing partial physical metadata on its own.
- The app should derive per-record totals from the supplied values:
  - upfront-only users:
    total record investment = upfront payment
  - upfront plus repayments:
    total record investment = upfront payment + (monthly repayment × duration)
  - repayments-only users:
    total record investment = monthly repayment × duration
- The app should derive outstanding scheduled repayments from each repayment
  schedule only:
  - remaining months
  - outstanding balance = remaining months × monthly repayment
- The product should not try to fully model refinance complexity in the first
  pass. If users later refinance or restructure, they can represent it as best
  they can within the available dated system-addition records.
- When history does not reach back to the investment date, finance surfaces
  should distinguish between:
  - value recovered from available recorded history
  - estimated total recovery since investment date
- Where more than one investment record exists, recovery and payoff surfaces
  should support:
  - whole-system totals
  - per-record interpretation where helpful
- Projection should be explicitly approximate and derived from historical
  savings behaviour, with wording that makes clear the estimate can change as
  more real usage data arrives.
- This means Solar Details should no longer be treated only as one global
  installation-level form. The important physical details should be captured
  at the same level as the dated addition/investment they belong to.
