# FE-008: Settings and Tariff Management

## Goal

Create the signed-in Settings experience where users complete and maintain the
non-provider data that powers savings, payback, reminders, and setup progress,
starting with tariff management as the first full vertical slice.

This feature should turn Settings into the product home for optional
user-managed data after `FE-007` has admitted the user and required provider
credentials have been connected.

For tariffs, this feature now explicitly pivots to a schedule-based model that
supports 30-minute accuracy across the week. The target user experience is
closer to the MyEnergi tariff editor: users define one or more price periods
and assign them across a weekly half-hour grid, while tariff versions remain
date-ranged for historical accuracy.

## Status

`Complete`

## Feature Completion Criteria

- A Settings area exists for signed-in users as the home for optional setup and
  ongoing user-maintained data
- Settings communicates setup-completion status clearly enough that users know
  what is still optional vs what unlocks additional product value
- Users can review the currently active tariff and historical tariff versions
- Users can add or edit tariff and contract details with clear recalculation
  impact
- Tariff versions support a weekly 30-minute import schedule rather than only a
  small fixed set of named windows
- Reminder and tariff-validity states are surfaced clearly
- Tariff management flows feel safe, understandable, and consistent with the
  rest of the app
- Admin "view as user" support works correctly across signed-in user-facing
  screens so operators can inspect Settings/Tariff behavior in the real user
  context
- The Settings information architecture is extensible enough that later
  optional setup domains such as finance, solar details, location, and
  notification preferences can plug in without reworking the whole section

## Known Issues / Follow-up

- **Schedule-based tariff not wired into cost calculation** —
  `apps/web/src/jobs/daily-summary.ts` derives cost bands using the legacy
  night/peak window model only. When a tariff version has a `weeklyScheduleJson`
  / price-period shape (including free-import periods), those rules are ignored
  and `daily_summaries` are computed with the wrong bands, propagating incorrect
  cost and savings figures into range history. This must be addressed as part of
  the real save story (P-043) or as a dedicated follow-up before schedule-based
  tariffs are used in production calculations.

## Notes

- This feature covers user-facing maintenance and setup-completion flows, not
  operator workflows.
- A small supporting bug-fix slice is included here for impersonation because
  broken "view as user" behavior blocks trustworthy testing of user-facing
  Settings flows.
- Tariff management is the first full implementation slice inside Settings and
  should remain the center of gravity for the feature.
- The older simple model of one night window plus one peak window is no longer
  the intended long-term tariff shape for this feature.
- Finance, solar details, location, and notification preferences may eventually
  live in Settings, but they should not bloat the first implementation pass
  unless they receive explicit story scope.
- The first concrete execution path for this feature is: establish Settings as a
  real product area, adopt the schedule-based tariff model, implement Tariffs as
  the first fully navigable section, then add the schedule editor and reminder
  states that make the section trustworthy.
