# Plan: U-021 — High-Fidelity Mockups for Core Signed-In Screens

## Overview

U-021 is a **design documentation story**, not a code story. The deliverable is a
detailed high-fidelity spec file (`docs/ui/hi-fi-signed-in.md`) that translates
the mid-fi layouts from `docs/ui/mid-fi-layouts.md` into concrete visual decisions
for the four core signed-in screens: Overview, Live, Daily History, and Range History.

Because no external design tool (Figma, Stitch) is yet established in this project,
the durable artifact format follows the existing pattern in `docs/ui/` — structured
markdown that is detailed enough for U-024, U-025, and U-026 to proceed without
re-deciding product structure.

## Ambiguities and Assumptions

### Ambiguity 1 — Design tool
The story is explicitly tool-agnostic. No Figma, Stitch, or similar tool link appears
in the repository. **Assumption:** The hi-fi output will be a structured markdown spec
with precise visual decisions described in plain language (color roles by semantic name,
type scale by relative size, spacing by rhythm descriptor). This follows the same
documentation-first pattern as `mid-fi-layouts.md` but at higher resolution.

### Ambiguity 2 — "High-fidelity" in a text artifact
Hi-fi typically means pixel-accurate visual files. In markdown it means: specific and
locked visual decisions rather than structural placeholders. **Assumption:** Hi-fi here
means committing to exact copy, named color roles (e.g. `surface`, `on-surface`,
`warning`, `muted`), type scale labels (e.g. `display`, `body-sm`), component names,
and specific layout proportions (column ratios, card internal padding rhythm). These
decisions are what U-025 will later formalize into tokens.

### Ambiguity 3 — Design-token language
U-025 owns token formalization. **Assumption:** U-021 may use descriptive proto-token
names (e.g. `color/warning`, `color/trust-positive`) as a vocabulary without binding
to any implementation format. This is distinct from "formal design tokens."

### Ambiguity 4 — Chart rendering detail
The story asks for "chart styling direction." Without visual files, this means describing
chart type, axes behavior, series color roles, tooltip behavior, empty/stale treatment,
and time-resolution toggle states in prose. **Assumption:** One chart-style spec block
per screen covers this requirement.

### Ambiguity 5 — State variant count
The story asks for "state variants where layout or trust presentation changes
materially." **Assumption:** For each screen, document the four states listed in the
story: healthy/default, setup-incomplete, stale/warning, and one error-adjacent state.
Not every permutation needs its own section — only states that change layout or module
presence.

### Step 3 — Write Visual Design Foundations section

Define the shared visual language that all four screens use. This section covers:

**Typography hierarchy:**
- Display: large KPI values and primary page answers
- Headline: section labels and card titles
- Body: explanatory text and helper copy
- Body-sm: metadata, timestamps, secondary notes
- Label: form labels, chart axis labels
- Caption: footnotes, caveats, source notes

**Color roles** (semantic, not hex):
- `surface` / `surface-raised` / `surface-inset`
- `on-surface` / `on-surface-muted`
- `brand-primary` — solar generation signal
- `positive` — savings, export, good states
- `warning` — stale, partial, setup-incomplete
- `destructive` — hard errors, disconnected provider
- `neutral` — grid, import, no-solar comparison
- `trust-positive` — data is fresh and reliable
- `locked` — gated modules and prompt cards

**Spacing rhythm:**
- Tight (4px base unit), regular (8px), comfortable (16px), loose (24–32px)
- Cards: comfortable internal padding; sections: loose separation

**Card anatomy:**
- Header area (label + optional status badge)
- Value area (primary metric, display size)
- Support area (secondary value, comparison, or trend direction)
- Footer area (optional: CTA, timestamp, action link)

**Chart styling direction:**
- One primary chart per screen section — no competing equal-weight charts
- Series colors: generation = brand-primary, consumption = neutral, import = warning-adjacent, export = positive
- Axes: minimal — time on x, value on y with unit suffix
- Grid lines: subtle, not dominant
- Tooltip: show all active series values at a time point
- Empty/stale treatment: desaturate + show timestamp and reason inline
- Time-resolution toggle: compact control inside the chart card, not in the page header

**Trust and warning states:**
- Trust badge: compact pill near the top of data-bearing screens; states: `fresh`, `stale`, `partial`, `disconnected`
- Warning banner: full-width below the header for screen-level issues; calm tone, specific cause, one CTA
- Inline warning: within a card when only one module is affected

**Locked-value / prompt-card anatomy:**
- Replaces the metric card it gates
- Same card dimensions as the metric it replaces to avoid layout shift
- Icon + locked label + one-sentence explanation + primary CTA
- CTA routes to the specific setup step that unlocks it

### Step 4 — Write Overview screen spec

**Rationale note:** Overview is the default signed-in landing. Its primary question is
"Am I saving money, and can I trust what I am seeing?" The hi-fi treatment must make
the period financial outcome immediately legible before the user reads anything else,
while surfacing trust/freshness as a secondary but unmissable signal. Provider-only
users (no tariff) must still land on a coherent, useful screen.

**Desktop hi-fi layout:**
1. Top nav bar — installation name, date range control (presets + custom), user menu
2. Data freshness / trust badge strip — trust badge + last-updated timestamp + optional warning link
3. Setup progress strip (conditional) — compact progress indicator + next-step CTA; hidden when setup is complete
4. Primary KPI row — 4 cards: Savings (or prompt), No-Solar Cost, Actual Cost, Export Value; 4-column grid
5. Payback card — full-width below KPI row if finance data present; prompt card if not
6. Two-column secondary row:
   - Left: Live snapshot card (current generation, import, export — compact)
   - Right: Interpretation panel (plain-language period summary)
7. Shortcut row — links to Live, yesterday in Daily History, Range History, Tariffs

**Mobile hi-fi layout:**
1. Top nav bar — minimal; installation name, date range icon, user avatar
2. Trust badge + last-updated (one line)
3. Setup progress card (conditional) — full-width, compact
4. Live snapshot card — full-width; current state for provider-only users
5. Primary KPI cards — stacked; savings first (or prompt), then no-solar, actual cost, export value
6. Payback card (or prompt) — full-width
7. Interpretation panel — full-width, collapsible on small screens
8. Shortcuts — horizontal scroll row

**State variants:**

*Healthy / default:*
- All four KPI cards show real values
- Trust badge = `fresh`, green-adjacent
- Payback card shows progress bar + projected date

*Setup-incomplete (no tariff):*
- Savings card replaced by locked prompt: "Add tariff details to see savings" + CTA
- Payback card replaced by locked prompt: "Add finance details to track payback" + CTA
- No-Solar Cost card remains visible as a standalone metric
- Live snapshot card still renders fully — provider-only experience is coherent
- Setup progress strip is visible and elevated

*Stale / warning:*
- Trust badge = `stale`; strip shows "Data last updated X hours ago"
- Warning banner below trust strip: "Live data appears delayed — historical summaries remain accurate" + "Review Data Health" CTA
- Financial KPIs remain visible using last-available data
- Live snapshot shows last-known values with elapsed-time stamp

*Hard error:*
- Financial modules replaced by a single failure card: "Unable to load your data" + cause summary + retry/reconnect CTA
- Trust badge = `disconnected`
- Navigation and Setup shortcuts remain accessible

### Step 5 — Write Live screen spec

**Rationale note:** Live's primary question is "What is my system doing right now, and
is this data fresh?" The hi-fi must emphasize realtime state — the four power-flow
metrics — before anything financially interpretive. Freshness must be prominent enough
that a stale state is immediately obvious, not something the user has to hunt for.

**Desktop hi-fi layout:**
1. Top nav bar — `Live` label, auto-refresh indicator, freshness timestamp
2. Trust badge / stale warning (elevated position — first visible signal after nav)
3. Current metrics row — 4 KPI cards: Generation, Consumption, Import, Export (W/kW values, large display text)
4. Solar coverage bar — horizontal visual showing % self-sufficient vs grid-reliant right now
5. Primary live trend chart — 30-minute rolling window; series: generation, consumption, import, export
6. Two-column secondary row:
   - Left: Current-day totals (kWh accumulations so far today)
   - Right: Warning / source notes (partial-day notice, provider notes, efficiency vs capacity if available)

**Mobile hi-fi layout:**
1. Top nav — `Live`, freshness timestamp
2. Trust badge / stale warning
3. 2×2 current metrics grid
4. Coverage bar (full-width)
5. Live trend chart (full-width, scrollable)
6. Current-day totals (stacked cards)
7. Warning / source notes

**State variants:**

*Healthy / default:*
- All four metrics live; freshness stamp = "Updated X seconds ago"
- Trend chart shows active data; coverage bar animated
- Efficiency card visible in secondary row if capacity data exists

*Setup-incomplete (provider-only, no capacity):*
- Efficiency metric silently omitted from secondary row — no warning, no empty state
- All four core metrics and chart render fully
- No setup prompt needed on this screen

*Stale / warning:*
- Trust badge = `stale`; freshness stamp = "Last seen X minutes ago" with warning color
- Metrics displayed with stale indicator overlay (desaturated + timestamp)
- Trend chart shows last-available data with "Data delayed" annotation on x-axis end
- Warning card in secondary row: specific cause + "Review Data Health" CTA

*Provider disconnected:*
- Replaces metrics with a centered connection-error card: provider name + failure description + "Reconnect" CTA
- Chart hidden; secondary row hidden
- Trust badge = `disconnected`

### Step 6 — Write Daily History screen spec

**Rationale note:** Daily History's primary question is "What happened on this day, and
what did it likely cost or save me?" The hi-fi must cleanly separate energy understanding
(what happened) from financial interpretation (what it cost). A user without tariff data
should still get full energy value from this screen. The day chart is secondary to the
summary rows — it explains rather than leads.

**Desktop hi-fi layout:**
1. Date header — date display, previous/next navigation, trust state badge
2. Energy totals row — 4 cards: Generation, Consumption, Import, Export (kWh); no financial values here
3. Financial comparison row — 3 cards: Actual Cost, No-Solar Cost, Savings; prompt cards when tariff absent
4. Day chart — generation and consumption as primary series; import/export as secondary; interval-level detail
5. Two-column secondary row:
   - Left: Day story / highlights (one-paragraph plain-language summary)
   - Right: Warnings (incomplete day notice, source notes, trust caveat)

**Mobile hi-fi layout:**
1. Date header with navigation
2. Energy total cards (stacked)
3. Financial cards or prompt cards (stacked)
4. Day chart (full-width)
5. Day story
6. Warning / notes

**State variants:**

*Healthy / default:*
- All six metric cards populated
- Day chart renders full interval data
- Day story gives one-sentence character of the day ("High generation, low grid reliance")

*Setup-incomplete (no tariff):*
- Energy totals row renders fully
- Financial row: all three cards replaced by a single prompt card spanning the row: "Add tariff details to see cost and savings for this day" + CTA
- Day chart renders on energy data only
- Day story omits financial commentary

*Partial day (today or mid-day):*
- Date header shows "(In progress)" label
- Energy cards show partial totals with "(so far)" suffix
- Financial cards show partial estimates with explicit caveat label
- Day chart shows filled portion + greyed-out future portion of the day

*Hard error / missing day:*
- Metric cards replaced by single error card: reason for absence (before install / no provider data / retrieval failure) + appropriate CTA
- Chart hidden

### Step 7 — Write Range History screen spec

**Rationale note:** Range History's primary question is "How am I doing over time, and
what pattern best explains the result?" The hi-fi must lead with period financial
outcome before any breakdown detail. The key takeaway sentence should be visible without
scrolling. The trend chart is explanatory — it comes after the KPI row and takeaway, not
before. Tariff-change flags and missing-data flags are tertiary, surfaced near the chart.

**Desktop hi-fi layout:**
1. Range header — preset controls (Last 7 days, Last 30 days, This month, Last 3 months, Custom); selected range label
2. KPI row — 4 cards: Total Savings (or prompt), Total Actual Cost, Total No-Solar Cost, Total Export Value
3. Key takeaway panel — one sentence period summary; full-width, moderately prominent
4. Primary trend chart — daily savings or daily generation over the period; single dominant chart
5. Two-column secondary row:
   - Left: Breakdown module (by day/week/month toggle; table or bar chart)
   - Right: Period flags (tariff-change events, missing-data gaps, explanation of exclusions)

**Mobile hi-fi layout:**
1. Range controls (horizontal preset row + "Custom" CTA)
2. KPI cards (stacked)
3. Key takeaway (full-width)
4. Trend chart (full-width)
5. Breakdown (full-width, toggle between views)
6. Period flags

**State variants:**

*Healthy / default:*
- All four KPI cards populated
- Key takeaway: e.g. "You saved €X over this period — your best equivalent of Y days"
- Trend chart shows daily savings bars; tariff-change flag inline on chart if applicable

*Setup-incomplete (no tariff):*
- Savings and no-solar cards replaced by a single prompt card: "Add tariff details to see financial performance over time" + CTA
- Actual Cost card also locked with same prompt
- Export Value card remains if available without tariff? — **Assumption:** Export value requires tariff (export rate); replace all financial cards with prompt.
- Key takeaway adjusts: "Energy overview for this period — add tariff details for financial analysis"
- Trend chart shows energy volume (generation/import) instead of savings; clearly labelled as energy-only

*Partial range (includes missing days):*
- KPI row shows totals with footnote: "X days excluded due to missing data"
- Period flags panel highlights gap dates
- Totals labelled as partial; trend chart shows gaps as breaks

*Stale:*
- Trust badge in range header: "Summaries may not include the most recent data"
- KPI row renders with last-available data; no values hidden
- Flag in period panel: "Latest sync: X hours ago"
