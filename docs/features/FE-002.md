# FE-002: Data Health and Coverage Trust

## Goal

Turn provider outages, missing periods, and suspicious telemetry into first-class
product data so the app can:

- warn users about current outages and anomalies
- report provider uptime and usable-data coverage over time
- show when savings or payback outputs are based on incomplete history
- support historical backfill from the installation date or best-known usable-history start
- notify users when meaningful outages or anomalies occur

This feature exists because current-day warnings alone are not enough. A provider
feed can be degraded or offline for days or weeks without the user noticing, which
can silently distort financial interpretation unless the product tracks and surfaces
data trust over time.

## Feature Completion Criteria

- The runtime model persists daily coverage/health summaries for each installation
- The runtime model persists outage/anomaly incidents with local start/end times and severity
- Historical provider backfill can derive and store coverage from installation date or a conservative usable-history boundary
- Data Health and related product surfaces can explain month/year uptime and missing-data impact clearly
- Savings/payback/reporting surfaces can disclose when outputs are based on partial history rather than silently implying full confidence
- Significant outages/anomalies can trigger user-facing notifications without duplicate alert spam

- Supplier-bill reconciliation as a user-facing product feature

## Notes

- Persisted outage/anomaly data should distinguish:
  - no provider data
  - partial provider data
  - suspicious or anomalous data
  - pre-install or outside-usable-history periods
  - later backfilled or recovered periods
- Historical coverage should feed user-visible trust language such as:
  - “Data coverage: 91% over the last 12 months”
  - “Based on 10.3 of 12 months of usable data”
- Financial and payback outputs should use this feature to avoid false precision when history is incomplete.
- The feature should be idempotent: re-fetching or backfilling a day should update persisted health/outage records rather than duplicating them.
