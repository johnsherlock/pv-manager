# P-053: Remove legacy tariff-window and free-import model residue

## Objective

Simplify the tariff schema and runtime code so the rewrite uses one tariff
shape only:

- date-ranged tariff versions
- user-defined price periods
- a 7-day × 48-slot weekly schedule

This story explicitly removes the older simple-window model and any special
"free import" treatment. A zero-cost import period should be represented only
by `rate_per_kwh = 0`.

## Scope

- Remove legacy tariff fields from `tariff_plan_versions`:
  - `day_rate`
  - `night_rate`
  - `peak_rate`
  - `night_start_local_time`
  - `night_end_local_time`
  - `peak_start_local_time`
  - `peak_end_local_time`
  - `free_import_rule_json`
- Remove the `is_free_import` field from `tariff_price_periods`
- Remove the `free_import_kwh` field from `daily_summaries`
- Remove the legacy compatibility path from tariff resolution and billing:
  - `src/domain/tariff-compat.ts`
  - legacy-window fallback branches in billing, daily-summary, range, live, and
    tariff loaders
- Remove placeholder writes that only exist to satisfy the old schema shape
  such as writing a fake `dayRate` during tariff saves
- Remove UI fallbacks that render tariffs as hard-coded Day / Night / Peak
  windows when no schedule exists
- Remove code and copy that treat "free import" as a first-class special case
  rather than a zero-rate period
- Update seed data and tests to use schedule-based tariffs only
- Update architecture, feature, and UI docs so they describe one tariff model
  consistently

## Cleanup Required

The following areas are expected to change as part of this story:

- `apps/web/src/db/schema.ts`
- `apps/web/drizzle/*` migration history/baseline
- `apps/web/src/domain/billing.ts`
- `apps/web/src/domain/tariff-compat.ts`
- `apps/web/src/jobs/daily-summary.ts`
- `apps/web/src/jobs/derive-summary.ts`
- `apps/web/src/range/*`
- `apps/web/src/live/*`
- `apps/web/src/tariffs/loader.ts`
- `apps/web/app/settings/tariffs/*`
- `apps/web/src/db/seed.ts`
- tariff, range, live, and daily-summary test suites
- docs that still mention `day/night/peak` or "free import" as model concepts

## Out of Scope

- Adding a new tariff UX concept beyond the current schedule-based editor
- Reworking no-solar baseline methodology beyond removing dependencies on the
  legacy tariff fields
- Historical data migration for production users outside whatever minimal local
  dev/seed reset is needed during pre-launch development

## Acceptance Criteria

- The runtime tariff schema contains only the flexible schedule-based model for
  import pricing
- A price period with `rate_per_kwh = 0` behaves correctly without any
  dedicated free-import flag or rule field
- Billing and daily-summary logic no longer depend on hard-coded Day / Night /
  Peak fields or window timestamps
- Tariff create/edit, overview, and history flows no longer read or write
  placeholder legacy tariff values
- Range and Live loaders no longer expose or depend on legacy tariff-window
  fields
- Seed data and automated tests cover schedule-only tariffs, including zero-rate
  periods
- Documentation no longer describes free import as a separate tariff concept or
  presents the old simple-window model as active
