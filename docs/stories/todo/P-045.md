# P-045: Make admin impersonation resolve the effective user correctly

## Objective

Fix the current admin impersonation bug where `View as user` sets an
`impersonating_user_id` cookie but the app continues to load data for the real
admin session because loaders still read `session.userId` directly.

This story should make impersonation work consistently across the signed-in app
by introducing one shared effective-user resolver and moving existing loaders
onto it.

## Scope

- Add a shared helper for resolving the effective user from request/session
  context
- Read the `impersonating_user_id` cookie when present
- Fall back cleanly to the real signed-in `session.userId` when no impersonation
  cookie is set
- Update loaders and shared server-side entry points that currently read
  `session.userId` directly so they use the effective-user helper instead
- Ensure impersonated navigation works for the main signed-in product surfaces:
  - Live
  - Historical Day / Range History
  - Settings
  - Tariffs
- Preserve the existing impersonation banner / exit path behavior

## Out of Scope

- No redesign of the admin console
- No new impersonation audit logging
- No role/permission system rewrite
- No user-switcher UX changes beyond whatever is required to make the existing
  flow work correctly

## Dependencies

- `P-041`

## Acceptance Criteria

- When an admin starts impersonation, the effective user resolves from the
  `impersonating_user_id` cookie instead of the admin's own `session.userId`
- When no impersonation cookie is present, the helper falls back to the real
  signed-in session user
- Shared loaders no longer duplicate impersonation-cookie lookup logic ad hoc;
  they use the common resolver instead
- The main signed-in app screens load the impersonated user's data rather than
  the admin's own data while impersonation is active
- Exiting impersonation returns the app to the admin's own data context cleanly
- The change does not break normal non-impersonated user access

## Notes

- Treat this as a shared server-side context fix, not a page-by-page one-off
- The key implementation target is a helper such as
  `resolveEffectiveUserId(session)` or equivalent request-aware variant
- The cookie being set but never read is the root bug; fixing only the admin
  entry point is not sufficient if downstream loaders still bind directly to
  `session.userId`
