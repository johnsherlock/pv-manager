# P-038: Capture and persist MyEnergi credentials during onboarding

## Objective

Create the rewrite-owned onboarding and persistence path for connecting a user's
MyEnergi account so backend jobs and APIs can read provider credentials from the
database-backed provider connection rather than from app-level environment
variables.

This story is the bridge between user setup and the new backend feature set:
without it, `P-036` and `P-037` would still depend on one shared deployment
secret instead of per-user provider credentials.

## Scope

- Define the rewrite-side onboarding step for entering MyEnergi credentials
- Persist one provider connection for the installation being onboarded
- Persist or reference the supplied MyEnergi username and password in a
  server-side-only credential store linked from `provider_connections`
- Define validation and failure behavior for:
  - missing credentials
  - invalid credentials
  - reconnect/update flows
- Ensure operators and normal product flows cannot read plaintext credentials
- Define the seeded development path for one deterministic provider connection
  using the developer-supplied test credentials outside committed docs/code

## Out of Scope

- No broader multi-provider connection UX
- No generalized secret-management abstraction for every future integration
- No user-facing tariff or finance setup work beyond what provider onboarding
  needs
- No historical backfill flow

## Expected Implementation

- Use the existing `provider_connections` row as the product-facing record for
  provider status and sync metadata
- Store credentials indirectly via a `credentialRef` or equivalent
  server-side-only linkage, rather than exposing them in normal query payloads
- During onboarding:
  - collect MyEnergi username
  - collect MyEnergi password
  - verify or safely test the connection server-side
  - persist the linked provider connection in active or recoverable-error state
- For development seed data:
  - create a deterministic seeded provider connection row
  - attach it to developer-provided MyEnergi test credentials through the same
    code path or storage pattern intended for production

## Constraints

- Backend MyEnergi access for v1 must read credentials per installation/user,
  not from one deployment-wide environment variable pair
- Plaintext credentials must not be returned to the browser, logs, or routine
  operator tooling
- The persistence model must support future credential rotation and reconnect
  flows without changing the installation or tariff model
- The development seed path may use provided test credentials, but those values
  should not be committed into product docs, code comments, or fixture files

## Dependencies

- `P-020`
- `P-012`
- `P-013`

## Acceptance Criteria

- The rewrite onboarding plan explicitly includes MyEnergi credential capture
- A provider connection can be persisted for one installation with a linked
  credential reference
- Backend stories can resolve provider credentials from the stored connection
  context rather than from deployment-wide env vars
- Invalid-credential and reconnect states are defined clearly enough for later
  implementation
- The development seed plan covers one deterministic provider connection using
  the supplied dev credentials without committing them into repo-tracked docs or
  code

## Tests / Verification

- Verify the planned storage shape is compatible with `provider_connections`
  and future reconnect flows
- Verify the onboarding plan keeps plaintext credentials off normal app payloads

## Notes

- This is an enabling onboarding/backend boundary story. It should land before
  or alongside any production cutover from the temporary V1 bridge to direct
  MyEnergi access.
