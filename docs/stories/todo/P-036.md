# P-036: Build the first direct MyEnergi day-detail API for the rewrite

## Objective

Replace the temporary V1 minute-data bridge with a rewrite-owned server-side
MyEnergi adapter that can return a normalized single-day payload for:

- today's Live view
- a requested historical day
- future backend consumers that need one canonical day-detail contract

This story is the point where the rewrite stops depending on the legacy proxy
for day-level reads. The output contract should stay app-owned and compatible
with the Live and Historical Day screens, but the upstream source should now be
MyEnergi itself rather than the V1 Lambda/API bridge.

## Scope

- Add a rewrite-owned MyEnergi adapter under `apps/web/src`
- Keep provider credentials, serial numbers, and upstream URL details
  server-side only
- Support fetching one requested local date for one installation/provider
  connection
- Normalize the upstream payload into the same kind of app-facing day-detail
  shape already established by `P-023`
- Resolve local-day query boundaries correctly for:
  - standard days
  - DST spring-forward days with a missing local hour
  - DST fall-back days with a repeated local hour
- Normalize timestamps into explicit canonical interval semantics rather than
  passing through ambiguous provider-local fields unchanged
- Surface structured upstream failure modes suitable for API consumers and later
  job reuse

## Out of Scope

- No persistence of interval rows in the rewrite database
- No nightly summary job orchestration
- No multi-day range API
- No auth or user-selection UI
- No provider support beyond MyEnergi

## Expected Implementation

- Add a provider module that:
  - accepts installation/provider-connection context plus one requested local
    date
  - resolves the linked MyEnergi credential reference for that installation's
    provider connection
  - derives the provider query window from the installation timezone
  - normalizes the response into explicit interval timestamps and app-owned
    field names
  - preserves minimal provider trace metadata for debugging
- Expose a route or server-side function that:
  - accepts a requested date
  - resolves the active installation/provider connection
  - calls the MyEnergi adapter
  - returns the normalized day-detail payload
- Reuse the existing rewrite normalizer contract from `P-023` where practical so
  the UI does not need a new payload family just because the upstream source has
  changed

Recommended output sections:

- `meta`
  - requested local date
  - installation timezone
  - source type
  - fetched-at timestamp
- `minuteData`
- `halfHourData`
- `hourData`
- `summary`
  - current metrics when the date is today
  - full-day totals
  - solar coverage metrics
- `health`
  - freshness
  - completeness
  - suspicious-reading or normalization warnings

## Constraints

- The browser must never call MyEnergi directly
- The rewrite must own the normalized response contract
- The implementation must resolve credentials from the database-backed provider
  connection context rather than from one deployment-wide env var pair
- Adapter code must treat the installation timezone as authoritative for
  deciding which local day is being requested
- DST handling must be solved in the adapter layer, not deferred to billing or
  UI code
- The implementation must not rely on the legacy V1 proxy's partially shifted
  `yr`/`mon`/`dom` fields

## Acceptance Criteria

- A server-side MyEnergi-backed day-detail path exists in `apps/web`
- The path can fetch today or a requested historical local date without using
  the temporary V1 bridge
- The path resolves MyEnergi credentials from the stored provider connection for
  the installation being queried
- The browser-facing result contains rewrite-owned field names rather than raw
  provider names
- The implementation produces consistent canonical timestamps for normal days
  and DST-edge days
- The returned shape remains sufficient for the existing Live and Historical Day
  consumers
- Provider auth failure, empty-day, and partial-day responses are surfaced
  clearly and distinctly

## Tests / Verification

- Add adapter-level tests for:
  - a normal day
  - a DST spring-forward day
  - a DST fall-back day
  - sparse or partial upstream payloads
- Add route-level verification that the top-level response shape is stable
- Verify the direct adapter does not reproduce the legacy DST-boundary bug
  documented in [`docs/architecture.md`](/Users/john/Documents/Projects/pv-manager/docs/architecture.md)

## Notes

- This story should prefer contract continuity over implementation continuity:
  matching the existing rewrite payload matters more than preserving legacy
  helper structure.
- The resulting adapter should be reusable by the nightly summary job rather
  than duplicating MyEnergi fetch and normalization logic there.
- The first implementation may assume seeded/dev installations are in
  `Europe/Dublin`, but the adapter must still treat DST transitions as
  first-class date-boundary cases.
