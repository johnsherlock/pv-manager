# P-035: Add a normalized weather and sun-context server slice for Live

## Objective

Define the server-side Live-context slice that will load persisted location
data, fetch forecast information, compute current solar/daylight values, and
return a normalized payload for the Live screen.

## Scope

- Define how geocoding is performed once and persisted
- Define forecast fetch cadence and cache TTL expectations
- Define the normalized return shape for Live weather and sun context
- Define local calculation of daylight remaining and current sun position
- Define graceful degradation rules when part of the external data is missing
- Define how the Live screen consumes this slice without runtime geocoding

## Out of Scope

- No actual route or service implementation
- No database migration work
- No client-side UI implementation
- No historical weather backfill

## Expected Implementation

- Document a server-side contract that:
  - loads installation location context
  - avoids per-request geocoding
  - fetches forecast data on a bounded cadence
  - returns 12-hour and 5-day forecast sections
  - returns sunrise, sunset, solar noon, daylight remaining, altitude, and azimuth
  - handles partial failures without collapsing the whole Live screen

## Constraints

- The normalized contract must be stable enough for `LiveScreen` wiring
- Geocoding must not run on every Live page load
- The design must support exact and approximate location modes
- Solar-context calculations must remain honest and sky-relative

## Dependencies

- `P-033`
- `P-034`

## Acceptance Criteria

- A clear server-side contract exists for Live weather and sun context
- Forecast cadence and cache TTL expectations are documented
- The contract avoids runtime geocoding on every page load
- Daylight-left and current sun-position calculations are documented as part of
  the server slice
- Graceful degradation behavior is documented for partial API failures

## Tests / Verification

- Verify the documented contract is sufficient for `LiveScreen` consumption
- Verify the contract supports both exact and approximate location modes

## Notes

- This is a server-slice design story, not an implementation story
- The goal is to normalize provider data early so UI work stays product-focused
