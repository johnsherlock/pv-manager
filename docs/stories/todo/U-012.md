# U-012: Surface current-day data health warnings

## Objective

Make the real Live and day-detail experiences trustworthy by surfacing clear,
calm warnings when current-day provider data appears stale, incomplete, or suspicious.

The prototype already demonstrates these states visually. This story turns them into
real product behavior driven by actual heuristics from the provider-backed day-detail payload.

## Scope

- Define and implement current-day health heuristics for the Live slice
- Surface health state in:
  - Live
  - the shared day-detail data path where relevant
- Cover at least these states:
  - healthy
  - stale
  - suspicious
  - empty / no live data yet
- Ensure the user gets:
  - visible status near the top
  - calm explanatory copy
  - a sensible next step where relevant

## Out of Scope

- No full Data Health screen implementation
- No email or push alerting
- No background job orchestration
- No historical-provider diagnostics beyond current-day trust cues

## Expected Implementation

- Define rewrite-owned health fields on the current-day payload, such as:
  - freshness timestamp
  - completeness flag
  - suspicious-reading indicators
  - summary state enum
- Use heuristics that can be implemented from the provider-backed minute data
- Wire those fields into the Live UI

Recommended starting heuristics:

- `stale`
  - last point is older than an agreed threshold for the current local time
- `suspicious`
  - one or more intervals spike abnormally relative to surrounding values
  - obvious outlier patterns in import / export / generation
- `empty`
  - request succeeds but returns no points

## Constraints

- Warning copy should be calm and credibility-building
- Avoid pretending to know root cause when only an anomaly is detectable
- The user should still see useful context where possible instead of a blank screen
- Health logic should live server-side or in shared domain code, not only in client presentation logic

## Dependencies

- `P-023`
- `U-023`
- `U-030`

## Acceptance Criteria

- Real Live data is accompanied by a computed health state
- The Live screen clearly distinguishes healthy, stale, suspicious, and empty cases
- Last-known values remain visible when appropriate in stale states
- Suspicious states explain uncertainty without claiming a definitive provider fault
- The app-facing payload exposes enough health metadata for the UI to render these states without reimplementing heuristics on the client

## Tests / Verification

- Add tests for at least:
  - stale detection
  - empty response handling
  - one suspicious/outlier case
- Manually verify the health states using controlled or simulated data where needed

## Notes

- This story exists to turn prototype trust cues into real behavior
- Keep the heuristic set simple and explicit for the first slice; it can expand later
