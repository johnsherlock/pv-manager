# F-007: Build golden fixtures

## Objective

Turn the current billing evidence and seeded sample data into a repeatable fixture set
with explicit expected outputs, so calculation behavior can be validated and held stable
 as the single-user app starts using real billing and tariff data paths.

## Scope

- Create fixture inputs for multiple billing periods
- Include expected outputs for:
  - import cost
  - fixed charges
  - export credit
  - net cost
  - savings
- Cover at least one tariff-transition case
- Keep fixtures small enough to inspect manually

## Out of Scope

- No runtime product feature work
- No UI implementation
- No provider-backed live data path

## Expected Implementation

- Add a fixture set under a repo-owned test/fixture location
- Use reduced or anonymized evidence where practical
- Include a short README or inline notes explaining:
  - source evidence
  - what each fixture is meant to validate

## Constraints

- Expected outputs must be explicit, not recomputed dynamically from the same implementation under test
- Prefer a small number of high-signal fixtures over a large opaque set
- Keep the fixtures aligned with the current calculation spec and seeded tariff examples

## Dependencies

- `D-004`
- `F-002`
- `F-003`
- `F-004`
- `F-005`
- `F-006`

## Acceptance Criteria

- Multiple billing fixtures exist with expected outputs
- At least one fixture spans a tariff version change
- Fixture intent is documented and understandable
- The fixture set is usable by parity and regression tests

## Tests / Verification

- Verify fixture values against known evidence and manual spot checks
- Confirm the fixture set can be consumed by `Q-002` and `Q-003`

## Notes

- This is validation infrastructure, not app functionality
- The seeded local app path should lean on this work once billing verification resumes
