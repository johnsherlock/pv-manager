# P-034: Design installation location model and geocoding persistence

## Objective

Define how installation location is captured, normalized, persisted, and reused
so the app can support privacy-aware Live weather and solar context without
geocoding on every page load.

## Scope

- Define exact and approximate location modes
- Define what raw user-entered location data is retained
- Define what canonical geocoded fields are persisted
- Define precision semantics for exact versus approximate location
- Define when geocoding runs and when it re-runs
- Define how missing, invalid, or stale location data affects Live behavior

## Out of Scope

- No implementation of DB schema or migrations
- No forecast-fetch implementation
- No Live UI implementation
- No paid or premium location features

## Expected Implementation

- Document schema-level requirements for installation location fields
- Document product rules for:
  - exact address versus approximate area capture
  - canonical latitude/longitude storage
  - precision-mode storage
  - location label/display value
  - provider metadata and last-geocoded timestamp
  - re-geocoding triggers after onboarding edits

## Constraints

- The model must support exact and approximate location in the initial feature
- Runtime Live rendering must not depend on geocoding per page request
- The design must preserve user privacy expectations and honest labeling
- The model should be provider-agnostic enough to support future provider swaps

## Dependencies

- `P-033`

## Acceptance Criteria

- Persisted location fields are defined clearly enough for later DB work
- Precision semantics are documented and distinguish exact from approximate location
- Re-geocoding triggers are documented
- Missing-location behavior is documented for Live
- Product rules are specific enough that onboarding and server-slice stories can
  proceed without reopening the data-model decision

## Tests / Verification

- Verify the documented model supports:
  - full address input
  - postcode/eircode/zipcode input
  - town/area-level input
  - later user edits to location

## Notes

- Approximate mode exists to preserve privacy while still enabling useful
  daylight and forecast context
- This story is about design requirements, not schema implementation
