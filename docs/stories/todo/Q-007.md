# Q-007: Define auth, approval, and provider-gate verification

## Objective

Define the verification coverage needed for the beta-access model so Google
sign-in, approval gating, provider validation, and admin-only actions are all
tested explicitly before beta users rely on them.

## Scope

- Define checks for first sign-in user creation
- Define checks for repeated sign-in while `awaiting_approval`
- Define checks for the approval transition to `approved`
- Define checks for suspended-user handling
- Define checks for provider validation failure and success
- Define checks for admin-only approval and "view app as user" behavior
- Define what can be manual vs automated in the first beta pass

## Out of Scope

- No public demo-mode validation plan
- No broader deployment or release workflow checks
- No supplier-bill or tariff-calculation validation

## Dependencies

- `P-013`
- `U-015`
- `P-040`
- `P-038`
- `U-048`

## Acceptance Criteria

- Verification coverage is documented for the main auth, approval, and provider
  gating transitions
- Manual and automated checks are separated clearly enough to execute before
  beta rollout
- The plan includes failure cases, not just happy paths
- Admin-only behaviors are covered explicitly rather than assumed

## Notes

- This is the confidence story for the first real-user access model
