# P-037: Build the Vercel-scheduled daily summary job for all active installations

## Objective

Create the rewrite-owned daily summarization path that runs just after each
installation's local midnight, fetches the finalized previous day from
MyEnergi, computes the persisted daily summary, and writes or updates one
`daily_summaries` row per installation and local date.

This is the backend job that turns provider day-detail data into the cheap,
queryable history model chosen for v1. It must be safe on Vercel, safe to retry,
and explicit about timezone handling.

## Scope

- Add a secure internal job entrypoint in `apps/web` for daily summarization
- Add the Vercel cron configuration needed to trigger the job automatically
- Enumerate all active provider connections/installations eligible for
  summarization
- For each eligible installation:
  - determine whether its previous local day is finalized and due
  - fetch the correct local day through the rewrite-owned MyEnergi adapter
  - derive the persisted daily summary fields
  - insert or update the matching `daily_summaries` row idempotently
- Record job-run outcome metadata and useful failure information for support and
  retries
- Define retry/idempotency behavior so repeated job invocations do not duplicate
  summary rows

## Out of Scope

- No broad historical backfill
- No outage/anomaly incident persistence beyond whatever the summary row already
  stores today
- No user-facing notification delivery
- No provider support beyond MyEnergi
- No Overview or Range History UI implementation

## Expected Implementation

- Add a scheduler-friendly job flow with pieces such as:
  - cron trigger
  - internal authenticated route or task handler
  - installation eligibility resolver
  - summary import/write service
- Use the installation timezone, not server timezone, to decide:
  - which local date should be summarized
  - whether enough post-midnight buffer has elapsed for the day to be treated as
    final
- For the initial Ireland-only beta, prefer one Vercel cron shortly after local
  midnight in `Europe/Dublin`, while still resolving the target summary date
  using installation-local time and DST-safe rules
- Persist or update enough run metadata to answer:
  - what date was attempted
  - for which installations
  - what succeeded
  - what failed and why

## Constraints

- The entrypoint must not be publicly callable without server-side
  authentication/secret validation
- The job must be idempotent per installation plus local date
- A retry must overwrite or safely update the intended summary row rather than
  creating duplicates
- DST changes must not cause the job to summarize the wrong local day
- The job must resolve MyEnergi credentials from each installation's stored
  provider connection rather than from one shared deployment env var pair
- The job should share MyEnergi fetch/normalization logic with `P-036` rather
  than implementing a second provider path
- Keep the design compatible with the low-cost Vercel hosting decision; do not
  introduce a separate always-on worker requirement

## Dependencies

- `P-020`
- `P-036`
- `P-006`

## Acceptance Criteria

- A secure scheduled summarization entrypoint exists in `apps/web`
- Vercel cron configuration exists and is documented well enough for deployment
- The job can determine which active installations need their previous local day
  summarized
- For the Ireland-only beta, one cron schedule shortly after local midnight is
  documented as the intended operational pattern
- The job writes or updates one `daily_summaries` row per installation plus
  local date
- Re-running the job for the same day is safe and does not create duplicate
  rows
- The job captures enough status information for operators to understand success
  and failure outcomes

## Tests / Verification

- Add unit or integration-style verification for:
  - installation eligibility by timezone
  - idempotent upsert behavior for an already summarized day
  - correct previous-day selection across DST transitions
- Add a manual or automated verification path for the authenticated internal
  entrypoint
- Verify a local development path exists to run the job without waiting for real
  cron execution

## Notes

- "After midnight" should be implemented as an installation-local rule, not as a
  naive UTC date check.
- For the first beta we can simplify operations by assuming all supported
  installations are in Ireland and running one cron shortly after local
  midnight GMT/BST, but the code should still resolve the previous local date
  via `Europe/Dublin` rather than by subtracting a fixed UTC day.
- If the first pass needs a small safety buffer after midnight to avoid querying
  a provider day before it is finalized, document that buffer explicitly and
  keep it consistent across the adapter and scheduler.
