# U-030: Build the first production UI slice in `apps/web`

## Objective

Turn the approved Live-screen prototype into the first real product slice in the
rewrite app, using seeded local data plus the temporary V1-backed minute-data path.

For this story, optimize for a working single-user app rather than for the full
onboarding/auth journey. A user should be able to run the app locally, load the
seeded installation context, and see a real Live screen backed by actual server-side
data rather than hardcoded prototype values.

## Scope

- Replace the current mock-driven Live screen data path with real server-side data
- Keep the first production slice focused on `Live`
- Support one seeded local user / installation path only
- Use:
  - seeded DB data for installation, tariff, and summary-backed financial context
  - provider-backed minute/day-detail data for the live energy layer
- Preserve the approved Live-screen structure and hierarchy from `U-023`
- Keep the page coherent when some optional data is unavailable

## Out of Scope

- No onboarding flow
- No signup, auth, invite, or user switching
- No full Overview implementation
- No Daily History or Range History implementation beyond links/placeholders if needed
- No production-grade provider account management UI

## Expected Implementation

- Refactor the existing Live screen in `apps/web/app/live`
- Move prototype-only hardcoded values behind a real server-side loader
- Read from:
  - seeded rewrite DB state for tariff/setup context
  - the `P-023` live/day-detail path for current-day minute data
- Render the actual values into the existing modules:
  - live metrics
  - chart
  - current-day totals
  - value panel when tariff context exists
  - solar coverage
  - trust/freshness states

Recommended implementation direction:

- keep the route server-rendered or server-data-backed
- keep provider calls on the server
- add clear loading / empty / error handling instead of falling back to fake data silently

## Single-User Assumptions

- One seeded installation is the active installation
- No login gate is required for the local-dev path
- Installation metadata and tariff context come from the seeded DB
- Current-day minute-level telemetry comes from the temporary V1-backed provider path

## Constraints

- Do not reintroduce mock data once the real path is available, except where a specific state variant still needs explicit simulation
- Keep the data flow inspectable and debuggable
- Prefer simple, explicit server-side composition over abstract architecture that slows the first working slice
- Reuse the approved prototype hierarchy unless real data exposes a concrete usability issue

## Dependencies

- `P-023`
- `P-024`
- `P-022`
- `U-023`
- `U-012`

## Acceptance Criteria

- The Live route in `apps/web` renders using real server-side data rather than only handcrafted mock data
- The page works for one seeded local user / installation without onboarding
- Current metrics are sourced from the provider-backed day-detail path
- Financial/value modules use seeded tariff and summary-backed context when available
- The page still handles:
  - missing tariff data
  - missing optional installation metadata
  - empty or failed provider reads
- The resulting screen remains recognizably aligned with the approved Live prototype

## Tests / Verification

- Add or update at least one test around the server-side data composition if practical
- Manually verify the main Live route against the seeded local setup
- Verify that fake prototype values are not still silently driving the main healthy/default state

## Notes

- This is the first real product slice, so correctness and inspectability matter more than completeness
- The goal is a working local single-user app path, not a launch-ready multi-user platform
- Once this story is done, Overview and history can build on a real Live foundation instead of a prototype-only one
