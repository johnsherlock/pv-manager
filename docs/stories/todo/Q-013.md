# Q-013: Add validation coverage for forecast, geocoding, and solar-context time logic

## Objective

Define the automated and manual validation coverage needed for Live weather and
solar-context functionality so time handling, location states, and partial API
failures are tested explicitly rather than assumed to work.

## Scope

- Define validation for:
  - DST transitions
  - installation timezone correctness
  - sunrise and sunset edge cases
  - daylight-remaining behavior after sunset
  - approximate-location labeling
  - partial API failures
  - geocode miss handling
  - cache behavior and stale-forecast handling

## Out of Scope

- No implementation of tests in this story doc
- No production feature code
- No billing or tariff validation

## Expected Implementation

- Document the required automated and manual checks for the feature
- Tie those checks to the server slice, onboarding/location states, and Live UI behavior
- Capture edge cases that are easy to miss in local happy-path testing

## Constraints

- Validation should cover both product behavior and time/coordinate correctness
- Tests should not assume exact-location-only behavior
- Coverage should be strong enough to catch timezone and caching regressions

## Dependencies

- `P-035`
- `U-039`

## Acceptance Criteria

- Required automated and manual checks are documented
- DST and timezone correctness are explicitly covered
- Approximate-location labeling and geocode failure handling are explicitly covered
- Cache behavior and partial API-failure behavior are explicitly covered
- Validation scope is specific enough for later test implementation

## Tests / Verification

- Review the documented checks against the feature stories to confirm that each
  important failure mode has an explicit verification path

## Notes

- The riskiest failures here are often time-related or partial-data-related, not
  simple rendering issues
