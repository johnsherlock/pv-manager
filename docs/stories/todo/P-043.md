# P-043: Build schedule-based tariff editor and contract reminder flows

## Objective

Implement the tariff create/edit experience in `apps/web` around a weekly
30-minute schedule, including price periods, validity window handling, contract
metadata, schedule validation, and honest recalculation-impact guidance.

## Scope

- Create tariff version flow
- Edit tariff version flow
- Price-period setup and rates
- Add/remove price-period behavior
- Add/remove tariff-scheme behavior
- Weekly half-hour schedule editing
- Validity window handling
- Contract metadata fields
- Schedule completeness / validation conflict behavior
- Save / discard behavior for dirty editor state
- Success messaging that explains recalculation impact
- Reminder-related UI hooks that reflect stale validity or review dates

## Out of Scope

- No advanced bulk tariff import
- No admin tariff tooling
- No unrelated settings sections

## Dependencies

- `U-028`
- `P-042`
- `P-044`

## Design Reference

- [`docs/ui/settings.md`](../../ui/settings.md) — Settings route structure
  and Tariffs section definition; editor routes should be nested under
  `/settings/tariffs` (e.g. `/settings/tariffs/new`, `/settings/tariffs/:id/edit`)
  (produced by U-049)

## Acceptance Criteria

- Users can create and edit tariff versions from the Tariffs section
- Users can assign one or more price periods across a weekly 30-minute schedule
- Users can add and remove price periods
- Users can add and remove tariff schemes / day-pattern groups
- Period labels are freeform, with suggested defaults such as `Day`, `Night`,
  `Peak`, and `Flat`
- A zero-cost period is modeled simply as rate `0.00`; no separate schema-level
  `free` concept is required
- Validation prevents incomplete or unsafe schedule definitions and clearly
  explains conflicts
- Validation covers the agreed editor rules from `U-027`:
  - every day of the week must belong to exactly one scheme
  - every half-hour slot within a scheme must be assigned to exactly one price
    period
  - price periods must not overlap on a given day
  - the full week must be accounted for before save
- Removing a price period or scheme is allowed during editing, even if the
  draft temporarily introduces gaps or leaves some days unassigned
- Save is the enforcement boundary:
  - save remains disabled when the draft is known to be incomplete, or
  - save shows clear validation errors if the user attempts to save an
    incomplete draft
- Validation feedback should make it obvious which days or half-hour slots are
  still unassigned
- Navigating away from the editor with unsaved changes warns the user and
  offers discard / cancel behavior
- Contract dates remain distinct from tariff validity in the UI and behavior
- Save-success messaging explains whether historical calculations may change
- Reminder-related states connect clearly to the corrective edit actions

## Notes

- This should feel like a trustworthy correction flow, not a raw data-entry
  form
- The implementation target is a MyEnergi-style half-hour schedule editor
- These validation rules are editor-time constraints; `P-042` only needs to
  render whatever valid data has already been stored
- Add/remove controls for price periods and schemes belong to the editor only;
  they should not appear on the overview page
- The "View" link for non-current tariff versions in the overview list
  (`page.tsx`) currently links to `#` — wire this up to a real read-only
  route or the edit route as part of this story
- When persisting a period with `ratePerKwh = 0`, decide whether to write
  `isFreeImport = true` in the DB (the billing layer uses it to track
  `free_import_kwh` separately) or collapse the distinction and remove the
  column — this must be resolved before the save lands in production
