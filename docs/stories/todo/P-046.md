# P-046: Wire schedule-based tariff into daily summary job

## Objective

Connect the existing schedule-based derivation logic in `derive-summary.ts` to
the daily summary job in `daily-summary.ts`. When a tariff version has a
`weeklyScheduleJson`, the job should use `deriveDailySummaryFieldsScheduled`
instead of `deriveDailySummaryFields`, and persist the resulting
`bandBreakdownJson` alongside the energy totals.

## Background

`P-044` introduced the schedule-based tariff schema and the domain logic for
schedule-based band derivation (`deriveDailySummaryFieldsScheduled`,
`deriveBandBreakdown`). `P-043` wired the editor to persist real schedule data.

The daily summary job has not been updated: `loadTariffWindowsForDate` reads
only the legacy night/peak window fields, and `upsertDailySummary` never writes
`bandBreakdownJson`. All installations using schedule-based tariffs therefore
produce incorrect band breakdowns and zero `freeImportKwh`.

## Scope

- Update the tariff loader in `daily-summary.ts` to detect whether the active
  tariff version has a `weeklyScheduleJson` and, if so, return the schedule and
  its price periods instead of the legacy window fields
- Update `summariseInstallation` to branch on which derivation path to use
- Update `upsertDailySummary` to write `bandBreakdownJson` when present
- Add or update tests in `jobs/__tests__/derive-summary.test.ts` covering the
  schedule-based path where not already covered

## Out of Scope

- Backfilling existing `daily_summaries` rows for installations that already
  have schedule-based tariffs (that is a separate catch-up operation)
- Changes to the range history or billing display layers (those already read
  `bandBreakdownJson` where present)
- Changes to the legacy window path (leave it working as-is)

## Dependencies

- `P-044` (schedule-based schema and domain logic — complete)
- `P-043` (real save of schedule data — complete)

## Acceptance Criteria

- When the active tariff version for a given date has a `weeklyScheduleJson`,
  the job calls `deriveDailySummaryFieldsScheduled` and the resulting
  `daily_summaries` row has a non-null `bandBreakdownJson`
- When the active tariff version has only legacy window fields (no
  `weeklyScheduleJson`), the existing `deriveDailySummaryFields` path is used
  unchanged
- When no tariff version exists for the date, both paths fall through to a
  summary with null band fields, unchanged from current behaviour
- The `freeImportKwh` column is populated correctly for schedule-based rows
- All existing tests continue to pass
- New or updated tests cover the schedule-based derivation path through the job

## Notes

- The domain logic (`deriveDailySummaryFieldsScheduled`, `deriveBandBreakdown`)
  already exists in `src/jobs/derive-summary.ts` and is tested — the story is
  primarily a wiring task in `daily-summary.ts`
- `bandBreakdownJson` is already a column in the `daily_summaries` schema;
  no migration is required
