# P-039: Add rate-band import breakdown to daily summaries

## Objective

Fix a gap in the daily summary model: when interval readings are summarised into
a `daily_summaries` row, the rate-band breakdown (day/peak/night) of total import
is currently discarded. This makes it impossible to calculate accurate time-of-use
billing from daily summaries alone, forcing the billing domain to apply the day
rate to all import as a known simplification.

The summarisation step is the only moment raw interval data is available (no
interval table is persisted in v1). The fix is to classify each minute reading
into its tariff band at write time and store the band totals alongside the
existing energy fields.

## Background

V1 handled this in `energy-calculator.ts` (`calculateTotals`): for each
half-hourly interval it checked `hour` and `dayOfWeek` against hardcoded window
constants and accumulated `dayImpTotal`, `nightImpTotal`, `peakImpTotal`,
`freeImpTotal`. The key difference in v2 is that tariff windows are stored
per-tariff in the database (`nightStartLocalTime`, `nightEndLocalTime`,
`peakStartLocalTime`, `peakEndLocalTime`) rather than hardcoded. The
`inWindow` helper in `src/domain/billing.ts` already handles midnight-crossing
windows correctly and can be reused.

## Scope

1. **Schema migration** — add four nullable columns to `daily_summaries`:
   - `day_import_kwh numeric(14,6)`
   - `night_import_kwh numeric(14,6)`
   - `peak_import_kwh numeric(14,6)`
   - `free_import_kwh numeric(14,6)` *(null for now — free-hours logic deferred)*

   All nullable so existing rows without band data remain valid.

2. **`deriveDailySummaryFields`** (`src/jobs/derive-summary.ts`) — add an
   optional `tariffWindows` parameter:
   ```ts
   type TariffWindows = {
     nightStartLocalTime: string; // e.g. '23:00'
     nightEndLocalTime: string;   // e.g. '08:00'
     peakStartLocalTime?: string | null;
     peakEndLocalTime?: string | null;
   };
   ```
   When provided, format each `MinuteReading`'s `hour`/`minute` as `'HH:MM'`
   and classify using `inWindow`. Accumulate `dayImportKwh`, `nightImportKwh`,
   `peakImportKwh` (and leave `freeImportKwh` null — deferred). When
   `tariffWindows` is null/absent, set all band fields to null (backward
   compatible with the existing call sites and tests).

   `DailySummaryFields` gains:
   ```ts
   dayImportKwh: number | null;
   nightImportKwh: number | null;
   peakImportKwh: number | null;
   freeImportKwh: number | null;
   ```

3. **Daily summary job** (`src/jobs/daily-summary.ts`) — load the active tariff
   version for the installation/date being summarised and pass its window fields
   to `deriveDailySummaryFields`. The job already loads installation context;
   extend it to also load the tariff.

4. **Billing domain** (`src/domain/billing.ts`) — update
   `DailySummaryForBilling` and `calculateBillingFromDailySummaries`:
   - Add optional band fields to `DailySummaryForBilling`:
     `dayImportKwh`, `nightImportKwh`, `peakImportKwh` (all `number | null`)
   - When band data is present: calculate actual import cost as
     `dayImport × dayRate + nightImport × nightRate + peakImport × peakRate`
   - When band data is absent (null): fall back to `totalImport × dayRate`
     (existing simplified behaviour)
   - The no-solar baseline continues to use day rate on the counterfactual
     import total — this is a documented simplification and is unchanged

5. **Range types** (`src/range/types.ts`) — add to `RangeSeriesDay`:
   ```ts
   dayImportKwh: number | null;
   nightImportKwh: number | null;
   peakImportKwh: number | null;
   ```
   Update the `note` field type:
   ```ts
   note: 'banded-daily-rate' | 'simplified-daily-rate';
   ```
   Emit `'banded-daily-rate'` when all billing days in the range have band data;
   fall back to `'simplified-daily-rate'` when any day lacks it.

6. **Range loader** (`src/range/loader.ts`) — include band columns in
   `loadDailySummaryRowsForRange`; extend `DailySummaryRow` with the four
   nullable band fields.

7. **Range billing** (`src/range/billing.ts`) — pass band fields through from
   loader rows to `DailySummaryForBilling`; expose them in `RangeSeriesDay`.

8. **Seed data** (`src/db/seed.ts`) — add representative day/peak/night splits
   to the seeded daily summary rows. Use approximate splits consistent with
   typical Irish October household usage and add `day_import_kwh`,
   `night_import_kwh`, `peak_import_kwh` to each seeded row. Values must sum to
   `import_kwh` for each day.

9. **Tests** — update:
   - `src/jobs/__tests__/derive-summary.test.ts`: add cases for band
     classification with and without tariff windows, verify midnight-crossing
     night window, verify peak window, verify fallback when no windows supplied
   - `src/range/__tests__/integration.test.ts`: update fixture rows with band
     data; assert that `series[n].dayImportKwh` etc. are present and that
     `summary.actual.importCost` reflects banded calculation; assert `note`
     is `'banded-daily-rate'`
   - `src/range/__tests__/billing.test.ts`: add cases for banded vs simplified
     billing path

## Out of Scope

- Free import / free Saturday hours — `free_import_kwh` column is added but
  null; classification logic deferred until `free_import_rule_json` is defined
- No-solar banded baseline — remains simplified (day rate on counterfactual total)
- Backfill of existing summary rows — rows written before this story will have
  null band columns and fall back to simplified billing; a future rebuild job can
  backfill if interval data is available

## Acceptance Criteria

- `daily_summaries` has `day_import_kwh`, `night_import_kwh`, `peak_import_kwh`,
  `free_import_kwh` columns (all nullable)
- `deriveDailySummaryFields` classifies minute readings into bands when tariff
  windows are supplied; band totals sum to total `importKwh`
- Daily summary job passes tariff windows to `deriveDailySummaryFields`
- `calculateBillingFromDailySummaries` uses banded rates when available, falls
  back to day rate when not
- Range API series includes `dayImportKwh`, `nightImportKwh`, `peakImportKwh`
  per day
- Range summary `note` reflects whether banded or simplified billing was applied
- Seed rows include explicit band splits that sum to total import
- All existing tests continue to pass; new band-classification tests added
