# P-022: Build the first summary-backed range-summary API

## Objective

Create the first rewrite-owned range-history backend API by loading persisted
`daily_summaries` and tariff data, resolving a requested local-date range, and
returning a structured range-summary payload for Overview and History use.

This story takes the earlier "billing read path" idea and turns it into the
actual backend contract needed by the product: comparison-first range totals,
trend-ready per-day rows, and clear completeness / tariff-change metadata.

## Scope

- Implement a server-side route or handler in `apps/web`
- Load:
  - installation and timezone context
  - tariff data including versioning
  - persisted daily summaries for the requested local-date range
- Validate the requested range and resolve it in installation-local date terms
- Invoke existing domain logic from `src/domain/billing.ts`, adapting it as
  needed for range-level summary outputs
- Return structured output suitable for:
  - range summary cards
  - trend charts
  - trust/completeness messaging
  - tariff-change indicators

## Out of Scope

- No direct provider API calls for range reads
- No UI implementation or page polish
- No authentication
- No supplier bill reconciliation
- No arbitrary interval-detail expansion beyond what `daily_summaries` already
  support

## Expected Implementation

- Add a route or server-side handler under `apps/web` that accepts a date range
- Fetch data via the existing DB layer
- Resolve the installation's local timezone before interpreting the range
- Call or extend the billing domain service to produce a stable range-summary
  response
- Return a payload aligned with [`docs/ui/data-contracts.md`](/Users/john/Documents/Projects/pv-manager/docs/ui/data-contracts.md),
  including sections such as:
  - `meta`
    - requested range
    - timezone
    - currency
    - generated-at timestamp
  - `summary`
    - actual cost
    - no-solar cost
    - savings
    - export value
    - range totals for generation / import / export / consumption
  - `series`
    - per-day totals for the requested period
    - trend-ready savings or cost values
  - `health`
    - completeness over the range
    - missing-day count
    - tariff-change flag
    - setup eligibility warnings such as missing tariff data

## Constraints

- Keep logic server-side only
- Reuse existing billing logic where practical instead of re-implementing calculations
- The route must interpret input dates in the installation timezone, not in
  naive server-local or browser-local terms
- Prefer a shape that can later support both page rendering and API-style
  consumption
- Missing summaries inside the requested range must be disclosed explicitly
  rather than silently treated as zero-usage days

## Acceptance Criteria

- A server-side range-summary path exists in `apps/web`
- The path validates and resolves requested ranges in installation-local date
  terms
- Billing logic executes successfully against persisted `daily_summaries`
- Output includes:
  - actual cost
  - no-solar cost
  - savings
  - export value
  - energy totals over the range
  - completeness and tariff-change metadata
- Output includes trend-ready per-day data suitable for later range charts
- The route works against seeded development data and remains compatible with
  production summaries written by the daily job

## Tests / Verification

- Add integration-style verification against seeded or fixture-backed summary
  rows
- Assert the returned output shape explicitly
- Add at least one test for a range spanning a tariff-version change
- Add at least one test for a range with missing summary days so completeness
  flags are validated

## Notes

- This is the first production range-history backend slice, not just a generic
  billing helper.
- It remains summary-first by design: the range path should stay cheap and fast
  by reading `daily_summaries` rather than replaying provider day-detail fetches
  across long periods.
- The requested range should be treated as inclusive local dates unless a later
  UI contract explicitly says otherwise.
