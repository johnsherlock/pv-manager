# U-053: Define system additions and multi-investment setup model

## Objective

Define the user-facing setup model that replaces the flat installation-level
finance profile with a list of dated system-addition records, so users can
represent the original install and later expansions/upgrades without forcing
all site economics into one record.

## Scope

- Define `system addition` as the primary user-managed record in Settings
- Define the first-pass structure of each record:
  - label / description
  - date
  - capacity added (`kWp`) when relevant
  - upfront payment
  - monthly repayment amount including interest
  - repayment duration in months
- Define how the system-addition list appears in Settings:
  - empty state
  - one-record state
  - multi-record state
  - add/edit/delete flow
- Define setup-completion rules for this domain
- Define the minimum finance details required for a record to be saveable in the
  first pass
- Define how deleting the last remaining record returns this domain to
  incomplete
- Define validation rules for the per-record payment fields
- Define the user-facing explanation of what this setup unlocks
- Define how existing single-record finance setups conceptually map into the
  new model without requiring historical perfection

## Out of Scope

- Detailed finance-account / lender object modeling
- Refinancing-specific workflows
- Batteries, inverters, or non-array hardware as separate subtypes unless the
  first UX contract truly needs them
- Downstream ROI/payback screen behavior beyond the requirements needed by
  `U-051`

## Acceptance Criteria

- The multi-investment/system-addition model is documented clearly enough to
  implement without falling back to one flat finance record
- `system addition` is established as the primary user-facing record name for
  the first pass
- The first-pass per-record fields are explicit and intentionally minimal
- Add/edit/list behavior is documented for desktop and mobile
- Completion rules explain when this domain counts as configured
- The first-pass model is explicit that a saved record includes finance details,
  rather than supporting physical-only records
- Delete behavior is explicit, including the last-record-to-incomplete case
- The relationship between physical addition details and payment details is
  clear to both users and implementers
