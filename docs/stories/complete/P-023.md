# P-023: Build the first provider-backed live/day-detail read path

## Objective

Create the first real live-data backend slice for the rewrite by exposing a
server-side path in `apps/web` that fetches single-day minute-level data from the
 existing V1 backend, normalizes it into a provider-agnostic shape, and returns
 data suitable for the Live screen and a future day-detail screen.

For this story, optimize for a working single-user app path rather than for a
fully abstract multi-user provider platform. The implementation should still keep
provider-specific details server-side and behind a stable app-facing shape.

## Scope

- Add a server-side read path in `apps/web` for one requested local date
- Use the existing V1 minute-data backend temporarily as the source of truth:
  - `https://jmcjm1731b.execute-api.eu-west-1.amazonaws.com/prod/minute-data?date=YYYY-MM-DD`
- Start with the same effective single-user assumption used by the seed data:
  - one installation
  - one provider source
  - no auth gating
- Normalize the V1 payload into a rewrite-owned day-detail shape
- Return enough data for:
  - current live metrics
  - minute trend series
  - half-hour and hour aggregations derived server-side
  - current-day totals
  - solar coverage metrics
  - trust/health metadata needed by `U-012`

## Out of Scope

- No authentication or user-selection flow
- No writeback or persistence of interval readings in the rewrite DB
- No long-range history endpoint
- No full provider abstraction for additional providers
- No background jobs, backfill, or cron orchestration
- No supplier-bill reconciliation in this story

## Expected Implementation

- Add a rewrite-owned provider module under `apps/web/src` for this temporary V1-backed source
- Keep the external URL and mapping logic server-side only
- Expose a route or server-side function that accepts:
  - `date`
- Normalize into a stable app-facing response with sections such as:
  - `meta`
    - requested date
    - timezone
    - source type
    - fetched-at timestamp
  - `minuteData`
  - `halfHourData`
  - `hourData`
  - `summary`
    - current metrics
    - today totals
    - solar coverage
  - `health`
    - freshness
    - completeness
    - suspicious-reading flags

Recommended file shape:

- `apps/web/src/live/` or `apps/web/src/providers/` for the temporary V1 adapter
- `apps/web/app/api/...` or an equivalent server-side module for the read path

## Temporary Source Contract

Use the provided endpoint directly for now, substituting the requested local date:

- Example for this story's reference date:
  - `https://jmcjm1731b.execute-api.eu-west-1.amazonaws.com/prod/minute-data?date=2026-03-28`

Implementation assumptions for this temporary source:

- The endpoint returns one JSON array for the requested day
- It may omit some fields on some records
- The response is minute-level but should still be treated as untrusted external input
- The rewrite must not leak raw V1 field names into the client-facing shape

## Constraints

- The browser must never call the V1 endpoint directly
- The rewrite must own the normalized response contract
- Reuse the V1 calculation and mapping logic where practical, but do not import UI-layer code
- Any code copied or adapted from V1 should be rewrite-owned and documented as temporary
- The returned shape should be sufficient for the current Live prototype without requiring mock data
- The story must work for the seeded single-user path even if broader auth/onboarding remains absent

## Acceptance Criteria

- A server-side live/day-detail read path exists in `apps/web`
- The path can fetch and normalize one requested day from the V1 minute-data endpoint
- The browser-facing result contains rewrite-owned field names rather than raw V1 payload keys
- The returned shape is enough to render:
  - live now metrics
  - recent trend chart
  - same-day totals
  - solar coverage
- Derived half-hour and hourly data are calculated server-side
- The route or module behaves sensibly when:
  - the upstream request fails
  - the upstream returns an empty array
  - the payload includes missing optional fields
