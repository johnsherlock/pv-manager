# U-046: Shared range picker with cross-screen navigation

## Objective

Extract the `RangePickerPopover` built in U-045 into a shared component and make
it available from the Live, History (daily), and Range History screens. The picker
becomes the single unified way to navigate across all time-based views.

Navigation is driven by the selection type:

| Selection | Destination |
|-----------|-------------|
| Single date — today | Live screen |
| Single date — any other day | History (daily) screen for that date |
| Any range (custom, week, month, year, all) | Range History screen |

---

## 1. Date tab (new)

A new **Date** tab is added as the first tab in the picker, before the existing
Weeks tab:

```
[ Date ]  [ Custom range ]  [ Weeks ]  [ Months ]  [ Years ]
```

- Same Su–Sa calendar grid as the Custom range and Weeks tabs.
- Single-click selects one date — no second click required.
- The selected cell is highlighted as a single filled circle (no range highlight).
- Selecting today navigates to the Live screen.
- Selecting any other date navigates to the History (daily) screen for that date.
- The preset strip is **not** shown on the Date tab.

---

## 2. Shared component extraction

`RangePickerPopover` moves from `apps/web/app/range/` to
`apps/web/src/components/RangePickerPopover.tsx` so it can be imported by any
screen.

The component's `onRangeChange` callback is replaced (or supplemented) by an
`onNavigate` callback:

```ts
type NavigationTarget =
  | { type: 'live' }
  | { type: 'history'; date: string }      // single date ≠ today
  | { type: 'range'; range: ActiveRange }  // any range selection
```

Each screen mounts the picker and handles `onNavigate` by pushing the appropriate
route:

| Screen | Live route | History route | Range route |
|--------|-----------|---------------|-------------|
| Live | stays / refreshes | `/history?date=YYYY-MM-DD` | `/range?from=…&to=…&mode=…` |
| History | `/` | stays / date param updated | `/range?from=…&to=…&mode=…` |
| Range History | `/` | `/history?date=YYYY-MM-DD` | stays / range state updated |

The Range History screen's internal `onRangeChange` flow (updating `activeRange`
without a full navigation) is preserved for range selections made while already on
that screen.

---

## 3. Period bar on Live and History screens

Both screens gain the same `[ < ] [ label ] [ > ]` period bar that Range History
has, allowing the picker to be opened and chevrons to step through dates.

- On the **Live screen**: the label shows "Today"; chevrons are disabled (there is
  no concept of stepping forward beyond today, and stepping backward navigates to
  History).
- On the **History screen**: the label shows the selected date; chevrons step
  one day at a time; stepping backward past `earliestDate` is disabled; stepping
  forward past today is disabled.

---

## 4. URL / state alignment

Range History should read `from`, `to`, and `mode` from URL search params so that
navigating from another screen with a range pre-selects it correctly, rather than
always defaulting to the last-30-days view.

---

## Acceptance Criteria

- [ ] A **Date** tab appears first in the picker on all screens
- [ ] Selecting today from any screen navigates to the Live screen
- [ ] Selecting a past date from any screen navigates to the History screen for
      that date
- [ ] Selecting any range (custom, week, month, year, all) from any screen
      navigates to Range History with that range pre-applied
- [ ] `RangePickerPopover` is in `src/components/` and not duplicated per-screen
- [ ] Live screen has the `[ < ] [ label ] [ > ]` period bar; label shows "Today";
      stepping backward navigates to History for yesterday
- [ ] History screen has the period bar; label shows the current date; chevrons
      step one day at a time
- [ ] Range History screen reads initial range from URL params

## Out of scope

- Animated page transitions between screens
- Remembering the last-used picker tab across sessions
- Deep-linking into Weeks/Months/Years tab selection from URL

## Dependencies

- `U-045` — RangePickerPopover implementation (complete)
- Live screen route must exist
- History (daily) screen route must exist and accept a `?date=` param
