# U-023: Build first polished clickable prototype for the Live screen

## Objective

Create the first polished clickable prototype for the Live screen so we can put a
realistic, reviewable version of the core live-data experience in front of people
 and learn from it quickly. This story should turn the existing low-fi, mid-fi,
 and one-off live-screen visual exploration into a concrete prototype that feels
 close enough to a product screen to evaluate.

## Scope

- Create a polished clickable prototype for the Live screen on:
  - desktop
  - mobile
- Translate the existing Live screen direction from:
  - `docs/ui/wireframes.md`
  - `docs/ui/mid-fi-layouts.md`
  - `docs/ui/data-contracts.md`
  - the one-off live-screen visual exploration from `U-021`
- Include the main live-view modules:
  - current metrics row or grid
  - freshness / health status
  - solar coverage / grid reliance treatment
  - primary live trend chart
  - current-day totals
  - warning / source notes area
- Include the most important prototype states:
  - healthy / default
  - stale data
  - warning / suspicious data
  - provider-only setup state where optional setup is incomplete
- Make the prototype clickable enough to review:
  - top-level nav state for `Live`
  - chart controls or time-resolution controls if shown
  - CTA or deep-link paths to Data Health and Daily History where relevant

## Out of Scope

- No Overview prototype work; that belongs to `U-022`
- No Daily History prototype work; that belongs to `U-024`
- No Range History prototype work; that belongs to `U-025`
- No onboarding or setup prototype work; that belongs to `U-026`
- No Tariffs, contract-management, or Data Health screen design beyond what is needed to reference them from the Live screen
- No design-system definition or token work
- No frontend implementation in `apps/web`
- No reopening of the structure already locked in `U-020`

## Expected Output

- One polished clickable desktop prototype for the Live screen
- One polished clickable mobile prototype for the Live screen
- Visual treatment for the key Live states listed above
- Enough clickability that a reviewer can understand:
  - what the current status is
  - what information is primary
  - where they would go next if the feed is stale or suspicious
- A short rationale note explaining how the prototype:
  - answers the "what is happening right now?" question
  - communicates trust and freshness
  - stays usable for provider-only users

## Constraints

- The Live screen must remain realtime-first, not savings-first
- Trust and freshness must be visible near the top of the screen
- The screen should be useful even when tariff and finance details are missing
- Missing installation capacity should hide efficiency gracefully rather than presenting an error
- The design should feel polished and product-like, but not overloaded with extra charts or dense dashboard treatment
- The primary chart should support understanding, not compete with the main live metrics
- Warning states should feel calm, credible, and actionable rather than alarming
- The prototype should use the existing live-screen exploration from `U-021` where helpful, but should not be blocked on preserving every detail from that exploration

## Acceptance Criteria

- A polished clickable prototype exists for the Live screen on desktop and mobile
- The prototype clearly shows:
  - current generation
  - current consumption
  - current import
  - current export
  - freshness / trust status
  - solar coverage or grid reliance
  - a primary live chart
  - current-day totals
- The screen remains coherent and useful for a provider-only user with optional setup still incomplete
- A stale-data state is represented in a way that is easy to notice and easy to understand
- A warning or suspicious-data state is represented with a clear next step
- The Live screen feels distinct from Overview and History by focusing on "right now"
- The output is strong enough to inform later implementation and adjacent clickable-screen stories without requiring another intermediate design step

## Review / Validation

- Review whether a user can understand the live state of their system in a few seconds
- Review whether freshness and trust are obvious without hunting for them
- Review whether the prototype still works if the user has only connected provider details
- Review whether the warning state encourages the right next action, such as checking Data Health
- Review whether the chart helps interpretation rather than adding noise
- Review whether desktop and mobile preserve the same hierarchy even though the layout differs

## Notes

- This is the first clickable prototype story in the streamlined workflow
- The goal is to get something realistic enough to review and react to, not to perfect the entire visual system in advance
- If the prototype reveals a structural issue in the Live screen, capture it explicitly so we can decide whether to adjust the shared screen model
- This story should create momentum for the other screen-specific prototype stories rather than creating a new layer of design process
