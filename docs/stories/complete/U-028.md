# U-028: Build polished clickable prototype for schedule-based tariff editing

## Objective

Create a polished, clickable tariff editor prototype that lets users add and
edit tariff entries using a weekly 30-minute schedule. The editor should feel
safe and corrective — not technical or back-office.

## Background and Design Decisions

The following decisions were made during design review and must be respected
in the implementation.

### One concept: a tariff entry with a date range

There is no separate "contract" concept in the UI. A tariff entry captures:
- who the supplier is
- what rates apply
- from when to when

If rates change mid-contract, the user sets an end date on the current tariff
and adds a new one from the date the new rates take effect. The app does not
need to know it was the "same contract". The reminder system (U-013) fires
when the end date approaches.

The `installation_contracts` table exists in the DB but is not surfaced as a
distinct concept in this UI.

### Editor information hierarchy

Capture in this order:

1. **Tariff identity** — supplier, optional plan nickname, start and end dates
2. **Charges** — export rate, VAT rate, standing charge
3. **Rate schedule** — weekly 30-minute schedule using the group/period model

This order reflects how a user thinks: who am I with, when does this run,
what do I pay, what are the actual rates.

### No version label field

The tariff is identified by supplier + date range (e.g. "Energia — 1 Jan 2026
to 31 Dec 2026"). There is no separate version label input. The label is
auto-generated from supplier and dates by the implementation layer.

### Rates valid to is required

"Rates valid to" defaults to 12 months after the start date. The user must
set an explicit end date — a blank end date is not permitted.

### All charge fields are mandatory

Export rate, VAT rate, and standing charge are all required. If a charge does
not apply, the user enters zero. This avoids nulls and ambiguity in
calculations.

### Standing charge: dual entry (annual ↔ daily)

Standing charges are typically quoted as annual amounts on supplier tariff
documents, but bills show a daily rate. The editor shows both fields and
auto-calculates the other when either is entered. Internally stored as per-day.

### Export rate is VAT-inclusive

Import rates and standing charges are entered and stored ex-VAT. The single
VAT rate is applied on top during calculation.

Export rates are different: they are paid to the user inclusive of VAT
(as seen on Energia bills where the export credit appears after the VAT
subtotal). Export rate is entered and stored as a gross (VAT-inclusive) figure
and must not have VAT added to it during calculation.

### VAT applies to: imports, standing charge, PSO levy (when modelled)

### Currency

Currency is already modelled at the `installations` level (`currency_code`,
`locale`, defaulting to `EUR` / `en-IE`). The editor UI currently hardcodes
`¢` for rate display; making this currency-aware is a follow-up task.

### PSO levy — deferred

The PSO (Public Service Obligation) levy is a fixed Irish government charge,
currently ~€1.46/month, appearing on bills before VAT. It is not modelled in
the current tariff schema. At ~€17.52/year the impact on calculations is small.

When added, it should be treated as an additional recurring monthly standing
charge subject to the standard VAT rate. Users outside Ireland may have
equivalent charges under different names. A future "additional recurring
charges" section should handle this generically rather than as an
Ireland-specific field.

## Schedule Editor Interaction Model

The editor's schedule section looks and feels like the existing Tariffs
overview display (the `TariffSchemeBlock` component), but made interactive.

### Schedule groups

A tariff entry contains one or more schedule groups. Each group defines which
days it applies to and what the per-period slot pattern is for those days.

Example: one group for Mon–Fri, one for Sat–Sun.

All days must be covered by exactly one group. All 48 half-hour slots within
each group must be assigned to a period. Only when these conditions are met
can the tariff be saved.

### Adding a group

The user clicks "+ Add tariff schedule". This adds a new group with no days
selected and no periods defined.

### Day pills

Each group has seven day pills (M T W T F S S). Toggling a day on adds it to
this group and removes it from any other group it was previously in (a day
belongs to exactly one group).

### Price periods

Each group has one or more price periods. Each period has:
- A name (text input, e.g. "Day", "Night", "Peak")
- A rate in €/kWh (numeric input; or a "free import" toggle for zero-rate
  periods like overnight free charging)
- A colour (auto-assigned from an 8-colour palette in order; not user-selectable
  in this prototype)
- A 48-slot activity bar (the editing surface)

The user clicks "+ Add price period" to add a row.

### Activity bar interaction

Each period's activity bar shows 48 slots. Slots assigned to this period are
filled with the period's colour; other slots are dimmed.

- **Click an empty slot** on period A's bar → assign that slot to A
  (automatically removes it from whichever period previously held it)
- **Click a filled slot** on period A's bar → remove the assignment
  (slot becomes unassigned)
- **Drag** across slots → paint or erase consistently based on the first
  slot's state at drag-start

There is no separate "erase" tool. The mutual-exclusion rule means assigning
a slot to one period always removes it from others — there are no overlaps.

### Validation

- All 7 days must be assigned to exactly one group
- All 48 slots in every group must be assigned to a period
- Every period must have a name and a rate (or free-import toggled on)
- Supplier name and date range are required
- All charge fields are required (0 is valid)

Save is blocked until all conditions are met. Errors are shown inline.

## Recalculation Impact Messaging

Saving a new or edited tariff entry will trigger recalculation of historical
cost and savings figures for all days within the entry's date range.

The save confirmation should explain this clearly:
- Which date range will be recalculated
- That this may take a few moments
- (Prototype only: note that no data was actually saved)

---

## Scope

- Create mode: blank form, rates valid to defaults to 12 months from today
- Edit mode: pre-filled from DB, including full weekly schedule
- Client-side validation with inline errors
- Prototype save: success confirmation (no real server mutation)
- "Edit" and "Add version" buttons in the Tariffs overview wired to real routes

## Out of Scope

- Real server-side save (that is P-043)
- PSO levy / additional recurring charges UI
- Currency-aware rate display (follow-up)
- Colour picker for periods (auto-assign for now)
- Admin-only controls

## Acceptance Criteria

- A polished clickable prototype exists for create and edit modes
- The editor captures tariff identity, charges, and rate schedule in that order
- The schedule editor uses the group/period/activity-bar model described above
- Day pills are toggleable; a day belongs to exactly one group
- Clicking a slot on a period's bar assigns it to that period (mutual exclusion)
- All validation states are surfaced clearly before save is permitted
- Save confirmation explains the recalculation scope
- "Edit" and "Add version" in the Tariffs overview link to the correct routes
