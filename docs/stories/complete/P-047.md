# P-047: Extend finance data model for flexible investment capture

## Objective

Implement the backend and persistence changes needed for the first Finance
settings slice so one installation can store a simple, user-editable solar
investment profile and derive coherent totals from a small set of flexible
finance inputs.

This story should support the UX defined in `U-050` while intentionally staying
with a single investment profile per installation.

## Scope

- Extend the installation finance model to support:
  - approximate investment date
  - upfront payment
  - monthly repayment amount
  - repayment duration in months
- Add the persistence and domain validation needed to enforce the agreed
  finance rules
- Derive:
  - total investment amount
  - remaining repayment months
  - outstanding scheduled balance
- Use one flexible finance profile rather than a stored payment-mode enum
- Expose a normalized finance settings payload for signed-in UI consumption
- Ensure setup-completion loading can tell whether Finance is configured
- Update seed data so local development can exercise the new model cleanly
- Add tests covering:
  - valid/invalid input combinations
  - derived total investment amount
  - derived remaining months / outstanding balance
  - incomplete state handling

## Out of Scope

- Multiple investment events
- Loan-provider metadata, APR breakdowns, or balloon payments
- Historical payoff calculations and projections in product UI

## Acceptance Criteria

- The app can persist one installation-level finance profile using the
  flexible model from `U-050`
- Invalid field combinations are rejected server-side
- Derived totals are deterministic from the stored inputs
- Settings loaders can distinguish configured vs incomplete finance state
- Seeded and test coverage exist for the agreed finance input combinations

## Dependencies

- `U-050`
- `P-020` (rewrite schema alignment)
- `P-021` (seeded installation finance context)
