# U-021: Create high-fidelity mockups for core signed-in screens

## Objective

Produce polished high-fidelity mockups for the main signed-in product surfaces so
the team can validate the visual direction before frontend implementation begins.
This story is the visual-design handoff from `U-020`, not a reopening of product
structure, IA, or data-contract decisions.

## Scope

- Create high-fidelity desktop and mobile mockups for:
  - Overview
  - Live
  - Daily History
  - Range History
- Define the visual treatment for:
  - typography hierarchy
  - spacing rhythm
  - card design
  - chart styling direction
  - trust and warning states
  - locked-value / prompt-card states
- Include state variants where layout or trust presentation changes materially:
  - healthy / default
  - setup-incomplete where relevant
  - stale / warning
  - one constrained or error-like state where needed
- Add a short rationale note per screen explaining how the mockup supports that
  screen's primary question and preserves the hierarchy defined in `U-020`

## Out of Scope

- No public-entry, signup, or onboarding high-fidelity work; that belongs to `U-022`
- No tariff, contract-management, or Data Health high-fidelity work; that belongs to `U-023`
- No clickable prototype assembly; that belongs to `U-024`
- No design-token or design-system formalization; that belongs to `U-025`
- No implementation-ready component specs; that belongs to `U-026`
- No frontend implementation work in `apps/web`; that belongs to `U-027`
- No changes to the layout hierarchy, module priority, or gating rules already locked in `U-020`

## Expected Output

- One high-fidelity desktop mockup for each in-scope screen
- One high-fidelity mobile mockup for each in-scope screen
- High-fidelity state variants for the most important non-default cases
- Short screen-level rationale notes covering:
  - the primary question the screen answers
  - how the visual hierarchy supports that question
  - how trust, freshness, and gated-value states are communicated
- Outputs may be produced in any design tool, but the resulting review artifacts
  must be stable enough to support `U-024`, `U-025`, and `U-026`

## Constraints

- Overview must remain summary-first and stay the default signed-in landing screen
- Live must remain realtime-first, not savings-first
- Daily History must clearly separate energy understanding from financial interpretation
- Range History must lead with period outcome before detailed breakdown
- Missing tariff data must show prompt cards, not fake or placeholder savings values
- Missing finance data must lock payback only, not the broader signed-in experience
- Missing installation capacity must hide efficiency gracefully rather than presenting an error
- Trust, freshness, and financial legibility take priority over dashboard density
- The four signed-in screens should feel visually related, but each must remain distinct in purpose
- High-fidelity work should translate `U-020`, not redesign it

## Dependencies

- `U-020`
- `docs/ui/mid-fi-layouts.md`
- `docs/ui/data-contracts.md`
- `docs/backlog.md`

## Acceptance Criteria

- High-fidelity desktop and mobile mockups exist for Overview, Live, Daily History, and Range History
- The mockups preserve the module hierarchy and interaction priorities defined in `docs/ui/mid-fi-layouts.md`
- Each screen clearly communicates its own primary question within a quick review
- Setup-incomplete, stale/warning, and other critical states feel intentional rather than broken
- Provider-only users still have a coherent signed-in experience in Overview and Live
- Savings and payback gating is visually clear, calm, and action-oriented
- The output is detailed enough for `U-024`, `U-025`, and `U-026` to proceed without re-deciding layout hierarchy
- The output stays aligned with `docs/ui/data-contracts.md` and does not invent conflicting backend expectations

## Review / Validation

- Review whether a new user with only provider details can still understand and use the Overview and Live experiences
- Review whether a user without tariff data sees helpful prompt cards instead of confusing or fake financial numbers
- Review whether a stale live-data state feels calm, credible, and actionable
- Review whether Daily History and Range History are immediately distinguishable in purpose and hierarchy
- Review whether the mockups trace cleanly back to `docs/ui/mid-fi-layouts.md` without introducing structural drift
- Review whether the visual direction is strong enough to extend into `U-022` and `U-023` without redesigning the product language from scratch

## Notes

- `U-021` is the first high-fidelity design story, but it should remain tool-agnostic
- If Stitch, Figma, or another design tool is used, the durable output still needs to be reviewable against this story and the `U-020` mid-fi spec
- If high-fidelity exploration reveals a structural problem, capture it explicitly for review rather than silently changing the product model
- This story should produce the visual baseline for later signed-in design work, not just four isolated mockups
