# P-033: Define free API provider contract for location, forecast, and sun context

## Objective

Choose and document the zero-cost provider approach for geocoding, forecast, and
sun-context data so later implementation work can proceed without reopening API
selection, pricing, or payload-shape decisions.

## Scope

- Compare zero-cost options for:
  - geocoding user-entered location data
  - hourly and daily forecast data
  - sunrise, sunset, and related sun-event data
- Record why Google Maps is out of scope for this feature
- Record why public Nominatim is not suitable as the primary app dependency
- Define the normalized product payload shape the Live screen needs
- Document preferred provider stack and acceptable fallback provider stack
- Document caching expectations, rate-limit expectations, and provider-specific risks

## Out of Scope

- No schema changes
- No onboarding copy or interaction design
- No runtime implementation
- No paid provider evaluation

## Expected Implementation

- Update product/architecture-facing docs with:
  - recommended free provider stack
  - fallback provider choice
  - required fields for geocoding, forecast, and sun-context modules
  - expected cache behavior
  - constraints and risks that later stories must respect

## Constraints

- Initial delivery must rely on zero-cost APIs only
- Provider choice must support both exact and approximate location capture
- The normalized contract should be product-driven, not provider-driven
- The decision should preserve the ability to swap providers later if usage or
  licensing constraints change

## Dependencies

- `U-031`

## Acceptance Criteria

- Docs name the preferred free provider stack
- Docs name at least one acceptable fallback provider path
- Docs explain why Google Maps is out of scope for this feature
- Docs explain why public Nominatim is not suitable as the primary dependency
- Required normalized fields are listed for:
  - location resolution
  - 12-hour forecast rail
  - 5-day forecast cards
  - sunrise/sunset and sun-position context
- Caching rules and provider-specific risks are documented

## Tests / Verification

- Verify the provider decision and payload contract are specific enough that
  implementation stories do not need to reopen the API selection

## Notes

- This is a decision-and-contract story, not an implementation story
- Open-Meteo and Geoapify are the main zero-cost candidates for the first pass
