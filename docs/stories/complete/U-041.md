# U-041: Range History page shell, period selector, and summary header

## Objective

Build the Range History page in `apps/web`: the route, the period selector, the
KPI summary header, and the trust/completeness strip. This story establishes the
page skeleton that all subsequent chart stories build on.

## Scope

- New page route (e.g. `app/history/range/page.tsx`)
- Server-side data loading: call `GET /api/range` with the selected date range,
  resolve the default preset (Last 30 days) on first load
- Period selector:
  - Named presets (Last 7 days, Last 30 days, Last 3 months, Last 12 months,
    This month, This year) as the primary control
  - Custom date range as a secondary path
  - Preset state preserved in URL query params so the page is shareable and
    browser-back works
- KPI row:
  - Savings € (solar savings for period)
  - Actual cost € (net cost after export credit)
  - No-solar cost € (what you would have paid without solar)
  - Export credit €
  - Avg solar coverage % (average `selfConsumptionRatio` across covered days)
- Tariff-change callout: when `health.hasTariffChange` is true, surface a
  dismissable notice near the KPI row naming the transition date(s)
- Completeness strip: when `health.completenessRatio < 1`, show missing day count
  and a brief explanation; when `health.partialDays > 0` add a partial-day note
- Trust / last-updated strip (consistent with other screens)
- All states from U-040: loading, empty, partial, no-tariff, hard error

## Out of Scope

- Charts (U-042, U-043, U-044)
- Free import display (deferred)

## Acceptance Criteria

- Page loads and renders for the default preset without errors
- Period selector changes trigger a fresh API call and re-render
- KPI values match the range API response fields
- Tariff-change callout appears only when `health.hasTariffChange` is true
- Completeness strip appears only when days are missing or partial
- No-tariff state replaces financial KPIs with a prompt card (energy totals still
  visible)
- Loading and error states render correctly
- Mobile and desktop layouts follow the direction from U-040

## Dependencies

- `P-039` (band data in range API — needed for accurate KPI values)
- `U-040` (screen direction — defines layout and state rules)
