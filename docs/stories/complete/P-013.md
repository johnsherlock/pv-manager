# P-013: Design Google auth, user lifecycle, and terms acceptance

## Objective

Define the first real beta auth model for the rewrite so users authenticate with
Google, the app manages approval state and roles, and no user passwords are
stored in the product database.

## Scope

- Define Google as the first supported auth provider
- Define the app-level user model for:
  - role (`user`, `admin`)
  - status (`awaiting_approval`, `approved`, `suspended`)
- Define the seeded-admin bootstrap approach using exact Google email
- Define how first sign-in creates a user record
- Define how the app treats "same email" identity, including Gmail plus-address
  variants
- Define where terms/privacy acceptance sits in the sign-in / first-use flow

## Out of Scope

- No email/password auth
- No Apple or other additional auth providers yet
- No public beta request flow
- No broader entitlement or premium-access model

## Dependencies

- `P-001`
- `P-007`

## Acceptance Criteria

- Google-only auth is documented as the initial beta auth approach
- The user model separates role from status clearly
- The admin seed path is defined using exact Google email identity
- The product rule for email identity is explicit: a different Google email is a
  different app user, and Gmail `+suffix` variants are not normalized away
- Terms/privacy acceptance expectations are clear enough for implementation

## Notes

- This story is the auth-model foundation for the rest of `FE-007`
- Identity, approval, and provider readiness should stay separate concerns in
  the design
