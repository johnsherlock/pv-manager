# P-044: Adopt schedule-based tariff schema and billing support

## Objective

Replace the simple tariff-window model with a date-ranged tariff version model
that supports a weekly 30-minute import schedule, so the product can represent
real-world tariffs such as weekday/weekend differences, EV windows, free-hour
offers, and flat-rate plans without relying on a small fixed set of named
windows.

## Scope

- Define the runtime tariff shape for:
  - dated tariff versions
  - one or more import price periods per version
  - a weekly 30-minute schedule assigning each half-hour slot to a price period
- Define the persistence approach, including migration away from or coexistence
  with the current `night_*` / `peak_*` window fields
- Update the billing/domain model so interval pricing resolves from the weekly
  schedule rather than only from one night window and one peak window
- Update summary-backed billing assumptions so persisted daily band/cost support
  remains coherent with the richer schedule model
- Define how flat-rate, day/night, day/night/peak, EV, and free-hours tariffs
  map into the new structure

## Out of Scope

- No final settings-screen UX
- No admin tariff tooling
- No bulk tariff import
- No production migration implementation details beyond what is needed to define
  the target model and safe transition path

## Dependencies

- `P-020`
- `P-019`
- `P-039`

## Acceptance Criteria

- A schedule-based tariff model is defined for weekly 30-minute accuracy
- The new model can represent flat-rate, day/night, day/night/peak, EV, and
  free-hours tariff patterns
- The runtime billing approach for both interval-backed and summary-backed
  calculations is updated to match the new model clearly enough for later
  implementation
- A migration/compatibility approach exists for the current simple window model
- The model keeps tariff validity dates separate from contract reminder fields

## Notes

- The UX target for this model is a MyEnergi-style weekly half-hour tariff
  editor
- 30-minute accuracy is sufficient; providers are not expected to need finer
  resolution than this
