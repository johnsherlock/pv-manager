# P-021 Implementation Plan: Seed Database with Summary-First Fixture Data

## Key Change from Previous Plan

The `installations` table in `schema.ts` is missing `finance_term_months` — it exists in
`docs/architecture.md` as `integer nullable` but was never added to the Drizzle schema.
P-021 must add this column before seeding.

The plan should also stay precise about what this seed can validate in v1: with
`daily_summaries`, it can exercise tariff-version date boundaries and payback
inputs, but it does not by itself validate day/night/peak interval pricing.

## Schema Change

Add one nullable integer column to the `installations` table in `schema.ts`:

```ts
financeTermMonths: integer('finance_term_months'),
```

This aligns the executable schema with `docs/architecture.md`, which already specifies
`finance_term_months integer nullable`. Run `npm run db:generate` after the change to produce
the `0001` migration. This is a non-breaking additive change — nullable column, no default required.

## Fixture Data Design

### User

- 1 record with a deterministic hardcoded UUID
- `status: 'active'`
- `auth_user_id`: deterministic UUID (no real Supabase auth user required locally)

### Installation

Seeded in `finance` mode to exercise the new `finance_term_months` field and keep
the “paid for itself” path unambiguous for a financed installation:

| Field | Value |
|-------|-------|
| `financeMode` | `'finance'` |
| `installCostAmount` | `null` |
| `monthlyFinancePaymentAmount` | `158.33` |
| `financeTermMonths` | `60` |
| `timezone` | `'Europe/Dublin'` |
| `locale` | `'en-IE'` |
| `currencyCode` | `'EUR'` |
| `providerType` | `'myenergi'` |

Notes:

- For this seeded installation, payback should be interpreted against the finance obligation
  (`monthlyFinancePaymentAmount × financeTermMonths`), not against an upfront cash cost.
- If a later story wants to exercise cash ownership too, add a second installation fixture
  rather than mixing ownership and finance inputs in one row.

### Provider Connection

- 1 myenergi connection, `status: 'active'`, linked to the seeded installation

### Tariff Plan

- Supplier: `Energia`, plan: `Smart Meter 24h`

### Tariff Plan Versions

Two versions spanning the September/October 2025 rate change observed in billing evidence
(`docs/calculation-spec.md`):

| Field | Version 1 | Version 2 |
|-------|-----------|-----------|
| `validFromLocalDate` | `2025-02-28` | `2025-10-10` |
| `validToLocalDate` | `2025-10-09` | `null` (open) |
| `dayRate` | `0.3451` | `0.3865` |
| `nightRate` | `0.1848` | `0.2125` |
| `peakRate` | `0.3617` | `0.4340` |
| `exportRate` | `0.2000` | `0.1850` |
| `vatRate` | `0.09` | `0.09` |
| Night window | `23:00–08:00` | `23:00–08:00` |
| Peak window | `17:00–19:00` | `17:00–19:00` |

### Fixed Charge Versions

Standing charge only per version. PSO levy (`per_month`) is omitted — the billing domain
handles only `per_day` in v1 and PSO can be added when that support lands.

| Version | `chargeType` | `amount` | `unit` | `validFromLocalDate` |
|---------|--------------|----------|--------|----------------------|
| Version 1 | `standing_charge` | `0.59` | `per_day` | `2025-02-28` |
| Version 2 | `standing_charge` | `0.66` | `per_day` | `2025-10-10` |

### Daily Summaries

14 days from `2025-10-03` to `2025-10-16`, crossing the `2025-10-10` tariff version boundary.
Values are manually crafted but realistic — varied generation, import, and export per day,
with some low-generation days to reflect typical autumn weather. `isPartial: false` on all rows.

This fixture validates:

- tariff version selection across a date boundary
- summary-backed aggregation inputs for the first billing slice
- installation payback inputs for financed installs

This fixture does not by itself validate:

- day/night/peak tariff-window attribution, because `daily_summaries` do not contain
  interval timestamps or time-of-use buckets

## Script Runner

Add `tsx` as a dev dependency. The npm script:

```json
"db:seed": "tsx src/db/seed.ts"
```

The seed script imports `db` and `pool` from `src/db/client.ts`, runs all inserts inside a
transaction using deterministic hardcoded UUIDs for every inserted row, applies
`onConflictDoNothing()` only where a real conflict target exists, then calls `pool.end()`.

## Ambiguities and Assumptions

| # | Item | Assumption |
|---|------|------------|
| 1 | **Migration generation** | `npm run db:generate` produces `0001` migration after schema change. Not hand-written. |
| 2 | **PSO fixed charge** | Omitted — billing domain handles only `per_day` in v1. Add later when `per_month` support lands. |
| 3 | **`auth_user_id`** | Set to a hardcoded deterministic UUID. No real Supabase auth user needed locally. |
| 4 | **Deterministic IDs** | Every seeded row gets a fixed UUID, including tariff plans, tariff versions, fixed charge versions, provider connections, and daily summaries. |
| 5 | **Idempotency** | Re-runs should avoid duplicates via deterministic primary keys and conflict-safe inserts, not by assuming every table has a natural unique key. |
| 6 | **Finance mode chosen over cash** | Seeding `finance` mode exercises both `monthlyFinancePaymentAmount` and the new `financeTermMonths`, giving P-022 enough to compute total finance obligation (`monthlyPayment × term`) for the "paid for itself" view without mixing in an upfront ownership cost. |

## Implementation Steps

1. Add `financeTermMonths: integer('finance_term_months')` to `installations` in `schema.ts`
2. Run `npm run db:generate` to produce the `0001` migration
3. Add `tsx` as a dev dependency; add `"db:seed": "tsx src/db/seed.ts"` to scripts in `package.json`
4. Create `apps/web/src/db/seed.ts` with deterministic fixture data covering all entities above, using fixed UUIDs for every inserted row
5. Apply the migration (`npm run db:push` or equivalent) and run `npm run db:seed` to verify no errors
