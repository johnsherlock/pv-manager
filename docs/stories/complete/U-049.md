# U-049: Define Settings shell and setup-completion home

## Objective

Define the signed-in Settings experience so users have one clear place to
complete and maintain optional product data after required provider setup is
done, without turning Settings into an unfocused dumping ground.

## Scope

- Define the Settings section list and navigation model
- Define the Settings home / shell for desktop and mobile
- Define setup-completion messaging:
  - what is optional
  - what unlocks additional product value
  - what is already complete
- Define how Tariffs appears as the first implemented settings domain
- Define how future sections such as finance, solar details, location, and
  notification preferences can appear without implying they are all fully built
- Define the relationship between Settings and provider reconnect/update entry
  points where relevant

## Out of Scope

- No admin/operator controls
- No public/demo surfaces
- No full Data Health section
- No detailed design for every future settings subsection

## Dependencies

- `U-001`
- `U-004`
- `U-020`

## Acceptance Criteria

- Settings information architecture is documented clearly enough to implement
  the shell
- Setup-completion messaging distinguishes optional setup from hard product
  gates
- Tariffs is clearly framed as the first real implemented settings domain
- Desktop and mobile navigation behavior is defined
- Future settings sections have a coherent home without forcing full scope now

## Notes

- This story should keep Settings focused and legible, not administrative or
  overloaded

### Status: Complete
